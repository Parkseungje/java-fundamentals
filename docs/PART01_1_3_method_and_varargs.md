# PART 1 — 객체지향(OOP) 기초: 1.3 메서드의 구조 + 가변인자

> 이 문서는 커리큘럼 PART 1의 소단원 중 **1.3 메서드의 구조 + 가변인자**만 다룬다.

---

## 0. 들어가기 전에 — 핵심 용어
- **메서드(method)**: 객체의 '행동'을 정의한 코드 블록(함수). 입력을 받아 동작하고 값을 돌려줄 수 있다.
- **시그니처(signature)**: 메서드를 식별하는 형태 — 이름 + 매개변수의 개수·타입·순서.
- **매개변수(parameter)**: 메서드 정의에 적힌 입력 변수. **인자(argument)**: 호출할 때 실제로 넘기는 값.
- **반환타입(return type)**: 돌려줄 값의 타입. 돌려줄 게 없으면 `void`.
- **접근제어자(access modifier)**: `public`/`private` 등 — 누가 이 메서드를 호출할 수 있는지(캡슐화).
- **가변인자(varargs, `타입... 이름`)**: 인자 개수를 0개~여러 개로 자유롭게 받는 문법. 내부에선 배열로 다뤄진다.

한 줄 그림: **메서드 = [접근제어자] [반환타입] 이름(매개변수) { 동작; return 값; }** 이고,
인자 개수가 들쭉날쭉하면 가변인자를 쓴다.

---

## 1. 학습 내용 — 메서드 시그니처와 가변인자

### 메서드 시그니처의 구성
메서드는 객체의 "행동"을 정의하는 코드 블록이며, 일반적인 형태는 다음과 같다.

```
[접근제어자] [반환타입] 메서드명(매개변수목록) {
    ... 동작 ...
    return 값;   // 반환타입이 void가 아니면 값을 돌려줘야 한다
}
```

각 요소의 역할:
- **접근제어자**(`public`/`private`/...): 누가 이 메서드를 호출할 수 있는지 결정. (캡슐화 — 1.1, 1.2와 연결)
- **반환타입**: 호출자에게 돌려줄 값의 타입. 돌려줄 게 없으면 `void`.
  반환타입이 있으면 `return 값;`으로 값을 돌려줘야 하고, 호출자는 그 값을 변수에 담아 쓸 수 있다.
  `void`면 돌려받을 값이 없으므로 변수에 담을 수 없다.
- **메서드명**: 동작을 나타내는 이름.
- **매개변수목록**: 호출 시 넘겨받을 입력. 0개일 수도 여러 개일 수도 있다.
  개수와 타입이 "호출할 때 무엇을 넘겨야 하는지"를 강제한다.

### 가변인자(Varargs)
때로는 "인자의 개수가 호출마다 다른" 경우가 있다. 예를 들어 로그를 찍는데 어떤 때는 메시지
하나, 어떤 때는 셋을 넘기고 싶다. 이를 위해 자바는 **가변인자(varargs)** 문법 `타입... 변수명`을 제공한다.

```java
void logAll(String prefix, String... messages) { ... }
```

핵심 규칙과 동작:
- 호출 측은 인자를 **0개부터 여러 개까지** 자유롭게 넘길 수 있다: `logAll("p")`, `logAll("p","a","b")`.
- 메서드 **내부에서는 배열(`String[]`)로 다뤄진다.** `messages.length`, `messages[i]`로 접근 가능.
- 가변인자는 **매개변수 목록의 마지막에만** 올 수 있다(그래야 어디까지가 가변인자인지 명확).
  그래서 위 예에서 고정 인자 `prefix`가 앞, 가변인자 `messages`가 뒤다.

### 자기 점검 미리 보기 — varargs vs 배열 매개변수
`void log(String... args)`와 `void log(String[] args)`는 메서드 내부에서는 **둘 다 `String[]`**로
똑같이 다뤄진다. 차이는 오직 **호출 편의성**이다. 가변인자는 인자를 콤마로 나열할 수 있고
(`log("a","b")`) 0개 호출도 가능한 반면, 배열 매개변수는 반드시 배열 객체를 만들어 넘겨야 한다
(`log(new String[]{"a","b"})`). 이 결론을 예시 3에서 코드로 확인한다.

## 2. 실습으로 확인하기

> - **가설 1**: 반환타입이 있으면 값을 받아 쓸 수 있고 void면 못 받는다. 매개변수 개수만큼 인자를 넘겨야 한다.
> - **가설 2**: 가변인자는 0개~여러 개 호출이 모두 되고, 내부에서는 배열(length/인덱스)로 다뤄진다.
> - **가설 3**: varargs와 배열 매개변수는 내부적으로 같지만, 호출하는 코드의 모양(편의성)이 다르다.

### 실행 결과 — 가설과 실제 비교

**예시 1 (시그니처)** — 가설 1.

| 메서드 | 시그니처 | 결과 | 가설 부합 |
|---|---|---|---|
| `format` | 반환 String, 매개변수 2개 | 반환값을 변수에 담아 `[INFO] 안녕하세요` 출력 | ✅ |
| `printLine` | 반환 void, 매개변수 1개 | 값을 못 받음, 호출 자체가 출력 동작 | ✅ |
| `answer` | 매개변수 0개, 반환 int | 인자 없이 호출 → `42` 반환 | ✅ |

**예시 2 (가변인자)** — 가설 2.

| 호출 | `messages.length` | 가설 부합 |
|---|---|---|
| `logAll("[A]")` | 0 | ✅ 0개 허용 |
| `logAll("[B]", "하나")` | 1 | ✅ |
| `logAll("[C]", "x","y","z")` | 3 | ✅ 내부 배열로 취급 |

→ 같은 메서드를 개수만 바꿔 호출해도 모두 동작하고, 내부 `length`가 0/1/3으로 찍힌다
= 가변인자가 내부에서 배열로 다뤄진다는 증거.

**예시 3 (varargs vs 배열 = 자기 점검)** — 가설 3.

| 호출 방식 | `logVarargs(String...)` | `logArray(String[])` |
|---|---|---|
| 콤마 나열 `("a","b","c")` | ✅ 가능 | ❌ 컴파일 에러 |
| 인자 0개 `()` | ✅ 가능 | ❌ 컴파일 에러 |
| 배열 통째로 `(new String[]{...})` | ✅ 가능 | ✅ 가능(유일한 방법) |

→ **자기 점검 답**: 둘 다 내부에서는 `String[]`로 같다. 하지만 가변인자는 "인자 나열 / 0개 /
배열" 모두 허용해 호출이 편하고, 배열 매개변수는 반드시 배열 객체를 만들어야 한다.
또한 둘은 컴파일된 시그니처가 사실상 동일(`String[]`)해서 **같은 이름으로 오버로딩할 수 없다**.

### 세 예시를 관통하는 결론
메서드는 "시그니처(예시1)로 호출 규약이 정해지는" 행동의 단위다. 가변인자(예시2)는 그 매개변수
규약을 "개수 유연하게" 확장한 문법적 편의이며, 그 실체는 배열이다(예시3). 즉 가변인자는
새로운 무언가가 아니라 **"배열을 더 편하게 넘기게 해주는 문법 설탕(syntactic sugar)"** 에 가깝다.

## 3. 자기 점검

- **Q. `void log(String... args)`와 `void log(String[] args)`의 차이는?**
  - 내 답: 내부 처리는 동일(`String[]`). 차이는 호출 편의성 — 가변인자는 인자 나열·0개·배열
    모두 가능, 배열 매개변수는 배열 객체만 가능. 시그니처가 같아 오버로딩 불가.
  - 근거 코드: `Example3_VarargsVsArray`의 호출 방식 비교 표.

- **Q. (추가 실험) 가변인자 메서드에 `null`을 넘기면? `logAll("[X]", (String[]) null)`**
  - 직접 실행해보고 `messages`가 빈 배열인지 `null`인지 확인한다. (가변인자에 명시적으로
    배열 null을 넘기면 `messages`가 null이 되어 `messages.length`에서 NPE가 날 수 있다 —
    가변인자라고 항상 안전하지 않다는 점을 확인)
