# PART 7 — 멀티스레딩과 동시성: 7.5 동기화 도구 3종 (synchronized / volatile / Atomic)

> 이 문서는 커리큘럼 PART 7의 소단원 중 **7.5 동기화 도구 3종**을 다룬다.
> (Atomic·CAS는 "9-섹션 깊이 학습" 권장) 7.4의 가시성·원자성 문제를 실제로 해결한다.

---

## 0. 들어가기 전에 — 핵심 용어
- **동기화(synchronization)**: 여러 스레드가 공유 자원에 '순서대로' 접근하게 통제해 동시성 문제(7.4)를 막는 것.
- **synchronized**: 한 번에 한 스레드만 들어가게 막는 키워드. 가시성+원자성 둘 다 해결(대신 느림).
- **모니터 락(monitor lock)**: 모든 객체가 하나씩 가진 '출입증'. synchronized 구역에 들어가려면 이 락을 얻어야 한다.
- **임계 구역(critical section)**: 한 번에 한 스레드만 실행해야 하는 코드 영역(공유 자원 변경부).
- **volatile**: 가시성만 보장(원자성 X). 한 스레드가 쓰고 여러 스레드가 읽는 플래그에 적합.
- **Atomic(AtomicInteger 등)**: 락 없이 원자적 연산을 제공하는 클래스. 내부적으로 CAS를 쓴다(빠름).
- **CAS(Compare-And-Swap)**: "값이 예상과 같으면 바꾸고, 다르면 재시도"하는 CPU 명령. 락 없이(논블로킹) 원자성을 얻는 원리.

한 줄 그림: **synchronized(락, 둘 다 해결·느림) → volatile(가시성만·빠름) → Atomic(CAS로 락 없이 원자성·빠름). 문제에 맞는 도구를 고른다.**

## 1. 학습 내용 — 세 도구의 진화

### 1-0. 출발점 — 7.4의 두 문제를 '어떻게' 막을 것인가
7.4에서 동시성 버그는 **가시성**(변경이 안 보임)과 **원자성**(다단계 연산이 끼어들어 덮어써짐) 둘이라고
했다. 7.5는 이 둘을 실제로 막는 도구 3종을 다룬다. 핵심은 **"어떤 도구가 어떤 문제를 막아주느냐"**가
다르다는 것 — 그래서 상황에 맞게 골라야 한다.

진화 서사로 보면 이해가 쉽다:
1. **synchronized** — 가장 확실하다. 한 번에 한 스레드만 들여보내 둘 다 막는다. 단 락을 잡고 기다려서 느리다.
2. **volatile** — "원자성까진 필요 없고 가시성만 싸게 해결하고 싶다"는 경우의 가벼운 도구.
3. **Atomic(CAS)** — "락 없이도 원자성을 빠르게" 얻으려는 진화. 멈추지 않고(논블로킹) 정확하다.

### 1-1. synchronized — 둘 다 해결(가시성+원자성), 대신 느림
**무엇**: 코드 블록이나 메서드에 붙여, **한 번에 한 스레드만** 그 안에 들어가게 막는 키워드.

**어떻게 동작하나 (모니터 락)**: 자바의 모든 객체는 **모니터 락(monitor lock, intrinsic lock)**이라는
'출입증' 하나를 갖는다. synchronized 구역에 들어가려면 그 객체의 락을 **획득**해야 하고, 나올 때 **반납**한다.
- 한 스레드가 락을 쥐고 있으면, 다른 스레드는 락이 풀릴 때까지 **BLOCKED 상태**로 대기한다(7.3 상태).
- 이렇게 "한 번에 하나만(상호 배제, mutual exclusion)" 들어가니, 임계 구역의 읽기-증가-쓰기가 **통째로**
  실행돼 **원자성**이 보장된다.
- 또한 락을 얻을 때/놓을 때 메모리를 메인 메모리와 맞추므로 **가시성**도 함께 보장된다. → 그래서 둘 다 해결.

```java
class Counter {
    private int count = 0;
    synchronized void increment() {   // 이 메서드 = this 객체의 락을 잡아야 진입
        count++;                       // 한 번에 한 스레드만 실행 -> 유실 없음
    }
}
```

**락의 대상이 무엇인가 (중요)**:
- `synchronized 인스턴스메서드` → 락 대상은 **그 객체(this)**.
- `static synchronized 메서드` → 락 대상은 **그 클래스의 Class 객체**(인스턴스와 무관, 클래스당 하나).
- `synchronized(obj){ ... }` 블록 → 락 대상은 **명시한 obj**. 보호가 필요한 부분만 감싸 **범위를 최소화**할 수 있다(락을 잡는 구간이 짧을수록 다른 스레드 대기가 줄어 빠르다).

**한계 3가지** (그래서 7.6 ReentrantLock이 등장):
1. **무한 대기** — 락을 얻을 때까지 무조건 기다린다(타임아웃을 못 건다).
2. **인터럽트 불가** — 대기 중인 스레드를 중간에 깨워 취소시킬 수 없다.
3. **공정성 보장 X** — 먼저 기다린 스레드가 먼저 얻는다는 보장이 없다.

> ★ 헷갈리는 지점 — "왜 느린가?" 락을 얻고 놓는 비용 + 락이 잡혀 있는 동안 다른 스레드가 **BLOCKED로
> 멈춰 기다리는**(일 못 하는) 비용 때문이다. 경쟁이 심할수록 대기가 길어져 느려진다.

### 1-2. volatile — 가시성만, 원자성은 ❌
**무엇**: 변수에 붙이는 키워드. "이 변수는 캐시에 가두지 말고 **항상 메인 메모리에서 직접 읽고 쓰라**"는 표시.

**무엇을 해결하나**: **가시성만** 해결한다(7.4 1-1). 한 스레드의 변경이 다른 스레드에 즉시 보인다.
그래서 7.4의 종료 플래그(`running`) 문제를 푼다.

**무엇을 못 하나**: **원자성은 못 해결한다.** `volatile int count; count++`는 읽기/쓰기 각각만 메인 메모리와
맞출 뿐, "읽기-증가-쓰기 3단계를 통째로 묶지" 못해 여전히 유실된다(7.4 1-3).

**언제 쓰나**: **한 스레드만 쓰고(write) 여러 스레드가 읽기(read)만 하는 플래그**. 대표 예:
```java
volatile boolean shutdown = false;   // 한 곳에서 true로 바꾸고, 여러 워커가 읽어 종료
```
이런 경우는 '증가' 같은 복합 연산이 없어 원자성이 필요 없고, 가시성만 있으면 충분하다. → volatile이 딱.

### 1-3. Atomic + CAS — 락 없이 원자성, 빠름
**무엇**: `AtomicInteger`, `AtomicLong` 등. 락을 잡지 않고도 `incrementAndGet()` 같은 연산을 **원자적**으로
해주는 클래스. 비결은 **CAS(Compare-And-Swap)**라는 CPU 명령이다.

**CAS의 원리 (핵심)**: "내가 본 값이 아직 그대로면 바꾸고, 누가 바꿨으면 다시 시도"한다.
```
CAS로 1 증가시키기:
  do {
    old  = 현재 값 읽기              // (A) 예: 100
    next = old + 1                  // (B) 101
  } while ( compareAndSet(old, next) 가 실패하면 )   // (C)
  // (C) "메모리가 아직 old(100)면 next(101)로 바꿔라"를 '한 번에(원자적)' 시도
  //     - 성공: 그 사이 아무도 안 건드림 -> 교체 완료
  //     - 실패: 그 사이 다른 스레드가 바꿈(이미 101) -> old를 다시 읽어 재시도
```
포인트: **락으로 남을 막는 게 아니라, "바뀌었으면 나만 다시 하면 된다"**는 방식이다. 그래서 다른 스레드를
멈춰 세우지 않는다(**논블로킹**) → 보통 synchronized보다 빠르다.

```java
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();   // 내부적으로 위 CAS 루프를 수행 (락 없이 원자적)
```

**주의 2가지**:
- **경쟁이 매우 심하면** CAS가 자꾸 실패해 재시도가 잦아질 수 있다(그래도 보통은 락보다 낫다).
- **ABA 문제**: 값이 A→B→A로 돌아오면 CAS는 "안 바뀌었다"고 오인한다. 값만 보고 "그대로네"라고 판단하기
  때문. 해결: `AtomicStampedReference`로 값에 '버전 도장'을 붙여 변화를 감지한다.

> ★ 헷갈리는 지점 — "락이 없는데 어떻게 안전하지?" CAS의 (C) 단계 자체가 CPU가 보장하는 **원자적 명령**이다.
> "비교와 교체를 쪼개지지 않게 한 번에" 수행하므로, 그 한 순간엔 끼어들 수 없다. 그 작은 원자 명령 위에
> "실패하면 재시도"를 얹어 전체 증가를 안전하게 만든 것이다.

### 1-4. 한눈 비교

| 도구 | 가시성 | 원자성 | 방식 | 멈춤? | 성능 |
|---|---|---|---|---|---|
| synchronized | ✅ | ✅ | 락(모니터) | 블로킹(BLOCKED 대기) | 가장 느림 |
| volatile | ✅ | ❌ | 메인 메모리 직접 R/W | 안 멈춤 | 빠름(단 원자성 X) |
| Atomic | ✅ | ✅ | CAS | 논블로킹(재시도) | 빠름 |

**선택 가이드**:
- 읽기 전용/단순 플래그(한 곳에서 쓰고 여럿이 읽음) → **volatile**
- 단일 변수의 원자 연산(카운터, 누적) → **Atomic**
- **여러 변수를 한 묶음으로** 보호하거나, 복합 로직(조건 검사 후 변경 등)의 원자성 → **synchronized**(또는 7.6 ReentrantLock)

선택 기준: 읽기 전용 플래그 → volatile, 단일 카운터/단일 변수 원자 연산 → Atomic, **여러 변수를 한
묶음으로** 보호하거나 복합 로직의 원자성이 필요 → synchronized(또는 7.6의 ReentrantLock).

## 3. 자기 점검

- **Q. synchronized/volatile/Atomic의 해결 범위와 성능을 비교하면?**
  - 내 답: synchronized=가시성+원자성/락/가장 느림, volatile=가시성만/메모리 동기화/빠름, Atomic=
    가시성+원자성/CAS 논블로킹/빠름. (Example3 표)

- **Q. CAS의 3단계와 ABA 문제는?**
  - 내 답: ①현재 값 읽기(A) ②새 값 계산(B) ③메모리가 아직 A면 B로 교체(아니면 재시도). ABA 문제는
    값이 A→B→A로 돌아왔을 때 "안 바뀐 것"으로 오인하는 것(AtomicStampedReference로 버전을 붙여 방지). (Example2)

- **Q. 단순 카운터에 synchronized보다 Atomic이 나은 이유는?**
  - 내 답: Atomic은 락을 안 잡고 CAS로 처리해(논블로킹) 대기가 없고 보통 더 빠르다. 단 단일 변수
    원자 연산에 한정 — 여러 변수를 한꺼번에 보호해야 하면 락(synchronized)이 필요하다. (Example3)

- **Q. synchronized 인스턴스 메서드와 static 메서드의 락 대상은 각각 무엇인가?**
  - 내 답: 인스턴스 메서드 = 그 객체(this), static 메서드 = 그 클래스의 Class 객체(클래스당 하나). 둘은
    서로 다른 락이라, 인스턴스 락을 잡아도 static 메서드는 못 막는다. (1-1)

- **Q. CAS는 락이 없는데 어떻게 안전한가?**
  - 내 답: "비교 후 교체(compare-and-swap)"가 CPU가 보장하는 원자적 명령이라 그 순간엔 끼어들 수 없다.
    그 위에 "실패 시 재시도"를 얹어 전체 연산을 안전하게 만든다(논블로킹). (1-3)

- **Q. volatile을 카운터(`count++`)에 쓰면 왜 여전히 틀리나?**
  - 내 답: volatile은 읽기/쓰기 각각의 가시성만 보장할 뿐, 읽기-증가-쓰기 3단계를 하나로 묶지 못한다.
    그 사이 다른 스레드가 끼어들면 유실된다. 원자성은 Atomic/synchronized로. (1-2, 7.4)
