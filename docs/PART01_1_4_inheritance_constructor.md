# PART 1 — 객체지향(OOP) 기초: 1.4 상속과 생성자 체이닝

> 이 문서는 커리큘럼 PART 1의 소단원 중 **1.4 상속과 생성자 체이닝**만 다룬다.

---

## 0. 들어가기 전에 — 핵심 용어
- **상속(inheritance)**: 한 클래스가 다른 클래스의 필드·메서드를 물려받는 것(`extends`).
- **부모/슈퍼클래스 ↔ 자식/서브클래스**: 물려주는 쪽 ↔ 물려받는 쪽. "개는 동물이다(is-a)" 관계.
- **생성자(constructor)**: 객체를 만들 때 초기화를 담당하는 특별한 메서드(클래스명과 같은 이름, 반환타입 없음).
- **기본 생성자(default constructor)**: 매개변수가 없는 생성자. 생성자를 하나도 안 쓰면 컴파일러가 자동으로 넣어준다.
- **`super(...)`**: 자식 생성자에서 부모 생성자를 호출하는 것. **생성자 체이닝** = 자식 생성 시 부모 생성자가 먼저 실행되는 연쇄.
- **`@Override`**: 부모 메서드를 자식이 재정의(오버라이딩)함을 표시.

한 줄 그림: **자식은 부모를 물려받고(extends), 객체를 만들면 '부모 부분 → 자식 부분' 순으로 초기화된다(super가 먼저).**

## 1. 학습 내용 — 상속, 생성자 체이닝, 그리고 다중 상속 금지

### 상속(Inheritance)이란
`extends` 키워드로 한 클래스(자식/서브클래스)가 다른 클래스(부모/슈퍼클래스)의 **필드와 메서드를
물려받는** 것이다. 자식은 부모의 것을 그대로 쓰면서, 자기만의 필드·메서드를 **추가**할 수 있다.

```java
public class Dog extends Animal {  // Animal의 name, eat()을 물려받음
    public void bark() { ... }     // Dog만의 행동 추가
}
```

상속의 목적은 **코드 재사용**과 **"is-a" 관계의 표현**이다. "개는 동물이다(Dog is an Animal)"라는
현실의 분류를 코드 구조로 옮긴다. 자식은 부모가 할 수 있는 모든 것을 할 수 있어야 한다
(이 약속이 깨지면 LSP 위반 — 1.8 SOLID에서 다시 다룬다).

### 생성자 체이닝 — 부모가 먼저 만들어진다
자식 객체를 생성하면, **자식 생성자는 본문을 실행하기 전에 가장 먼저 부모 생성자를 호출한다.**
이를 명시적으로 쓰면 `super(...)`이고, 안 쓰면 컴파일러가 자식 생성자 첫 줄에 매개변수 없는
`super()`를 **자동으로 삽입**한다.

왜 부모가 먼저인가? 자식은 부모의 필드를 물려받아 사용하므로, 자식 부분이 초기화되기 전에
부모 부분이 먼저 완성되어 있어야 안전하기 때문이다. 그래서 객체는 항상
**"부모 부분 → 자식 부분"** 순서로 초기화된다.

### 함정 — 부모에 기본 생성자가 없을 때
컴파일러가 자동 삽입하는 것은 **매개변수 없는** `super()`다. 그런데 부모 클래스가
**매개변수 있는 생성자만** 가지고 있으면(기본 생성자가 없으면), 자동 삽입된 `super()`가
부를 대상이 없어 **컴파일 에러**가 난다.

> 주의: 클래스에 생성자를 하나도 안 적으면 컴파일러가 기본 생성자를 자동으로 넣어주지만,
> 매개변수 생성자를 **하나라도 직접 정의하면** 자동 기본 생성자는 사라진다. 그래서 이 함정이 생긴다.

#### 왜 매개변수 생성자를 만들면 기본 생성자를 자동으로 안 넣어줄까?
자동 기본 생성자는 어디까지나 **편의 장치**다. 생성자를 하나도 안 쓴 클래스도 객체는 만들 수
있도록, 자바가 매개변수 없는 빈 생성자를 대신 넣어주는 것뿐이다.

그런데 개발자가 `Account(String owner)`처럼 생성자를 **직접 정의했다는 것**은,
"이 객체는 owner 없이는 만들어선 안 된다"는 **생성 규칙(의도)** 을 코드로 선언한 것이다.
이때 자바가 매개변수 없는 `Account()`를 몰래 추가해 버리면:

- owner가 `null`인 **반쪽짜리 객체**가 만들어질 수 있고,
- 개발자가 세운 불변식(invariant — "owner는 항상 있어야 한다")이 조용히 깨진다.

그래서 자바는 **"개발자가 생성 방법을 직접 정했다면 그 의도를 존중하고, 편의용 기본 생성자는
끼워 넣지 않는다"** 는 원칙을 따른다. 즉 자동 기본 생성자는 "개발자가 생성에 대해 아무 말도
하지 않았을 때만" 제공되는 것이다. 기본 생성자가 필요하면 개발자가 `public Account() {}`를
명시적으로 추가해야 한다(그러면 매개변수 생성자와 함께 둘 다 존재하게 된다).

해결: 자식 생성자에서 **명시적으로 `super(인자)`** 를 호출해, 어떤 부모 생성자를 쓸지 지정한다.
`super(...)`는 반드시 자식 생성자의 **첫 줄**이어야 한다.
(또는 부모에 기본 생성자가 꼭 필요하다면 부모 클래스에 `public Account() {}`를 직접 추가한다.)

### 다중 상속 금지와 다이아몬드 문제
자바는 클래스의 **다중 상속을 금지**한다(한 클래스는 하나의 클래스만 `extends` 가능).
이유는 **다이아몬드 문제(diamond problem)** 다.

```
      A (method foo())
     / \
    B   C   (둘 다 foo()를 다르게 재정의)
     \ /
      D   (B와 C를 동시에 상속한다면?)
```

만약 D가 B와 C를 동시에 상속할 수 있다면, `d.foo()`를 호출할 때 B의 foo()와 C의 foo() 중
무엇을 실행해야 할지 **모호**해진다. 이 충돌을 피하기 위해 자바는 클래스 다중 상속을 막았다.
대신 인터페이스로 여러 타입을 구현하게 하여 "능력의 조합"은 허용한다(이 보완은 1.6 인터페이스에서 다룬다).

## 2. 실습으로 확인하기

> - **가설 1**: 자식은 부모의 필드/메서드를 물려받아, 자식에 정의하지 않은 메서드도 호출할 수 있다.
> - **가설 2**: 자식 생성 시 부모 생성자가 자식 생성자보다 먼저 실행된다(부모 → 자식 순).
> - **가설 3**: 부모에 기본 생성자가 없으면 자식은 명시적 `super(...)`가 없을 때 컴파일 에러가 난다.

### 모델 코드 (`com.study.part01_oop.s04_inheritance_constructor`)
- `Animal`(부모) / `Dog`(자식) — 예시 1·2에서 사용. 생성자에 출력문을 심어 실행 순서 관찰.
- `Account`(부모, 매개변수 생성자만 있음) / `SavingsAccount`(자식, 올바르게 `super` 호출) — 예시 3에서 사용.

### 실행 결과 — 가설과 실제 비교

**예시 1 (상속)** — 가설 1.

| 호출 | 정의 위치 | 결과 | 가설 부합 |
|---|---|---|---|
| `dog.eat()` | Dog엔 없음, Animal에서 물려받음 | `초코가 먹는다` | ✅ |
| `dog.bark()` | Dog가 추가 | `초코: 멍멍!` | ✅ |

→ Dog 클래스에 `eat()`를 한 줄도 안 적었는데 호출된다 = 상속으로 물려받았다는 증거.

**예시 2 (생성자 체이닝)** — 가설 2. `new Dog("초코")` 시 출력 순서:

```
[Animal 생성자] name=초코 초기화     <- 부모가 먼저
[Dog 생성자] 초코 준비 완료          <- 그다음 자식
```

→ 부모 생성자가 자식 생성자보다 먼저 실행됐다. 객체는 "부모 부분 → 자식 부분" 순으로 초기화된다. ✅

**예시 3 (super 함정)** — 가설 3.

| 케이스 | 결과 |
|---|---|
| `SavingsAccount`(super(owner) 명시) | ✅ 정상 컴파일·실행 → `[Account 생성자]` → `[SavingsAccount 생성자]` |
| (가상) super 생략한 자식 | ❌ 컴파일 에러: `there is no default constructor available in Account` |

→ super() 자동 호출은 **부모에 기본 생성자가 있을 때만** 통한다. 부모가 매개변수 생성자만
가지면 자식이 직접 `super(인자)`로 어떤 부모 생성자를 부를지 지정해야 한다. Example2처럼 항상
자동으로 되는 게 아니라는 점이 핵심 차이다.

### 세 예시를 관통하는 결론
상속은 "부모 것을 물려받고(예시1) + 부모를 먼저 만든 뒤(예시2) + 자식을 완성한다"는 흐름이다.
생성자 체이닝은 이 "부모 먼저" 원칙을 보장하는 장치이며, 부모에 기본 생성자가 없으면 그 연결을
개발자가 `super(...)`로 직접 이어줘야 한다(예시3). 그리고 이 "한 부모"의 연결을 자바가 **하나로
제한**하는 이유가 다중 상속의 다이아몬드 문제다.
