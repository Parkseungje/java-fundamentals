## 1. 학습 내용 — 클래스와 객체는 무엇이고, 왜 구분하는가

### 클래스 = 청사진, 객체 = 실물
객체지향에서 가장 먼저 익혀야 할 구분은 **클래스(class)** 와 **객체(object, 인스턴스)** 의 차이다.

- **클래스**는 "이런 것이 어떻게 생겼고 무엇을 할 수 있는지"를 정의한 **설계도(청사진)** 다.
  흔한 비유로 **붕어빵 틀**이다. 틀 자체는 먹을 수 없다 — 그저 "붕어빵은 이렇게 생겼다"는 형틀일 뿐이다.
- **객체(인스턴스)** 는 그 클래스를 바탕으로 `new`로 찍어낸 **실제 물건**이다.
  붕어빵 틀로 찍어낸 **실제 붕어빵**이다. 틀은 하나라도 붕어빵은 여러 개 찍을 수 있고,
  각 붕어빵은 속(팥/슈크림)이 다를 수 있다.

이 비유의 핵심은 **"틀은 하나, 실물은 여럿이고 각자 다르다"** 는 점이다. 즉 같은 클래스로
만든 객체들은 같은 *구조*를 갖지만 *상태(데이터)는 제각각* 독립적이다.

### 클래스가 가져야 할 두 가지: 상태 + 행동
하나의 클래스는 보통 두 가지를 함께 담는다.

1. **상태(필드)** — "이 객체가 지금 어떤 상황인가"를 나타내는 데이터.
   계좌라면 주인 이름과 잔액, 자동차라면 속도 같은 것.
2. **행동(메서드)** — "이 객체로 무엇을 할 수 있는가".
   계좌라면 입금/출금, 자동차라면 가속/감속.

1.1에서 본 "상태와 행동을 하나로 묶는다"가 바로 이것이다. 그리고 중요한 점은,
**행동은 항상 "자기 자신의" 상태에 작용한다**는 것이다. `account.deposit(1000)`은
"그 account 객체의" 잔액을 늘린다. 같은 메서드라도 어느 객체에 대고 호출하느냐에 따라
바뀌는 대상이 달라진다 — 행동은 떠다니는 함수가 아니라 "특정 객체에 소속된" 동작이다.

### 자기 점검 질문 미리 보기 — 상태 없이 행동만 있는 클래스?
그렇다면 상태(필드) 없이 행동(메서드)만 있는 클래스는 의미가 있을까? 답을 먼저 말하면
**"의미는 있지만, 객체를 여러 개 만들 이유는 없다"** 이다. 이자 계산기처럼 "입력을 받아
결과를 돌려주기만 하고 기억할 것이 없는" 행동의 묶음이 그 예다. 이 결론을 예시 3에서
코드로 직접 확인한다.

## 2. 실습으로 확인하기

위 이론이 맞다면 다음 세 가지 가설을 세울 수 있고, 각각을 예시 하나로 검증한다.

> - **가설 1**: 같은 클래스로 만든 객체들은 상태를 공유하지 않고 각자 독립적으로 가진다.
> - **가설 2**: 같은 메서드 호출이라도 대상 객체의 상태에 따라 결과가 다르다(행동은 자기 상태에 묶여 있다).
> - **가설 3**: 상태 없는 클래스도 행동의 묶음으로 쓸모는 있으나, 인스턴스를 여럿 만들 이유는 없다.

### 모델 코드 (`com.study.part01_oop.s02_class_and_object`)
- `BankAccount` — 상태(owner, balance) + 행동(deposit, withdraw)을 가진 전형적 클래스. 예시 1·2에서 사용.
- `InterestCalculator` — 필드가 하나도 없는, 행동만 있는 클래스. 예시 3에서 사용.

| `Example3_StatelessClass` | 상태 없는 클래스는 의미가 있나? | 계산기 인스턴스 2개 + static 호출 비교 |

### 실행 결과 — 가설과 실제 비교

**예시 1 (틀 vs 실물)** — 가설 1: 객체마다 상태는 독립적이다.

| 동작 | 홍길동 계좌 | 김철수 계좌 | 가설 부합 |
|---|---|---|---|
| 초기 | 1000 | 1000 | — |
| 홍길동에만 5000 입금 | **6000** | **1000 (그대로)** | ✅ 독립 |

→ 같은 클래스로 만들었지만 한쪽 입금이 다른 쪽에 전혀 영향을 주지 않는다.
클래스는 틀일 뿐, 실제 데이터는 객체마다 따로 존재한다(각 객체가 힙에 자기 필드 공간을 가짐 — PART 2 심화).

**예시 2 (상태에 묶인 행동)** — 가설 2: 같은 메서드도 대상 객체 상태에 따라 결과가 다르다.

| 호출 | 홍길동(5000) | 김철수(2000) | 가설 부합 |
|---|---|---|---|
| `withdraw(3000)` | 성공 → **2000** | 잔액 부족 → **거절(2000 유지)** | ✅ |

→ 완전히 같은 코드(`withdraw(3000)`)인데 결과가 갈린다. 메서드가 "호출 대상 객체의
상태(`this.balance`)"를 기준으로 판단하기 때문이다. 행동은 특정 객체에 묶여 있다.

**예시 3 (상태 없는 클래스 = 자기 점검)** — 가설 3: 쓸모는 있으나 인스턴스를 여럿 만들 이유는 없다.

| 확인 | 결과 | 해석 |
|---|---|---|
| `calc1.calculate(10000, 0.03)` | 300 | 행동은 정상 작동 → 쓸모 있음 |
| `calc2.calculate(10000, 0.03)` | 300 | 다른 인스턴스인데 결과 동일 |
| `r1 == r2` | `true` | 상태가 없으니 인스턴스를 구별할 이유가 없음 |
| `InterestCalculator.calculateStatic(...)` | 300 | 그래서 객체 없이 static으로 호출하는 게 자연스러움 |

→ **자기 점검 답**: 상태 없는 클래스도 "행동의 묶음(유틸리티/계산기)"으로서 의미가 있다.
다만 "상태가 없다 = 객체마다 다를 게 없다"이므로 인스턴스를 여러 개 만들 이유가 없다.
이것이 상태를 가진 `BankAccount`(객체마다 다른 잔액)와의 결정적 차이다.

### 세 예시를 관통하는 결론
객체지향의 기본 단위인 "객체"는 **(예시1) 독립된 상태** 와 **(예시2) 그 상태에 묶인 행동**의
결합이다. 이 둘이 있어야 "객체마다 의미가 다른" 인스턴스가 성립한다. 반대로 (예시3) 상태가
없으면 행동만 남아 모든 인스턴스가 동일해지고, 그럴 땐 객체를 만들 필요 없이 static/싱글톤이
자연스럽다. 즉 **"인스턴스를 굳이 만들 필요가 있는가?"** 의 답은 결국 **"기억할 상태가 있는가?"** 에 달려 있다.

## 3. 자기 점검

- **Q. 상태 없이 행동만 있는 클래스는 의미가 있을까?**
  - 내 답: 의미는 있다(유틸리티·계산기처럼 상태가 필요 없는 행동 묶음). 다만 상태가 없으면
    인스턴스를 여럿 만들 이유가 없으므로 static 메서드나 싱글톤이 적합하다.
  - 근거 코드: `Example3_StatelessClass`에서 `r1 == r2`가 `true` → 인스턴스를 구별할 수 없음.

- **Q. (추가) `BankAccount`를 static 필드/메서드로만 만들면 어떻게 될까?**
  - `balance`를 static으로 바꾸면 "모든 계좌가 잔액 하나를 공유"하게 된다. 직접 실험해
    홍길동 입금이 김철수 잔액까지 바꿔버리는지 확인하고, 왜 "객체별 상태"에는 인스턴스 필드가
    필요한지 정리해본다. (static vs 인스턴스 필드의 차이 — PART 2 메모리 구조와 연결)
