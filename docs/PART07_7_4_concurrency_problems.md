# PART 7 — 멀티스레딩과 동시성: 7.4 동시성 문제 2가지 — 가시성과 원자성

> 이 문서는 커리큘럼 PART 7의 소단원 중 **7.4 동시성 문제(가시성·원자성)**를 다룬다.
> PART 2.3/7.1에서 본 "공유 변수 race"의 정체를 가시성·원자성 두 문제로 정확히 분해한다.

---

## 0. 들어가기 전에 — 핵심 용어
- **동시성 문제**: 여러 스레드가 공유 데이터를 같이 다룰 때 생기는 버그. 크게 가시성·원자성 두 갈래.
- **공유 변수**: 여러 스레드가 함께 접근하는 변수(인스턴스/static — Heap/Method Area). 지역 변수(Stack)는 안전(7.1).
- **가시성(visibility)**: 한 스레드가 바꾼 값이 다른 스레드에 '보이는가'. 안 보이면 옛 값을 읽는 문제.
- **CPU 캐시 / JIT 최적화**: 성능을 위해 각 코어가 값을 캐시하거나, 루프 밖으로 값을 빼두는 것. → 가시성 문제의 원인.
- **원자성(atomicity)**: 어떤 연산이 '쪼개지지 않고 한 번에' 일어나는가. `count++`는 읽기-증가-쓰기 3단계라 원자적이지 않다.
- **경쟁 상태(race condition) / 갱신 유실(lost update)**: 동시 수정으로 결과가 꼬이는 상황 / 한쪽 증가가 다른 쪽에 덮여 사라지는 것.
- **volatile**: 가시성을 보장하는 키워드(항상 메인 메모리에서 R/W). 단 원자성은 보장 못 함.

한 줄 그림: **동시성 버그는 '가시성(변경이 안 보임)'과 '원자성(동시 수정이 덮어씀)' 둘이다. 별개의 문제이고, volatile은 가시성만 해결한다(7.5에서 원자성까지).**

## 1. 학습 내용 — 두 가지 별개의 문제

### 1-0. 먼저 토대 — 왜 동시성 '문제'가 생기는가 (메모리 모델)
7.1에서 "스레드는 Heap·static 변수를 공유한다"고 했다. 여러 스레드가 같은 변수를 동시에 만지면 왜
탈이 날까? 핵심은 **컴퓨터가 "성능을 위해" 두 가지 일을 하기 때문**이다.

#### ★ 먼저 짚기 — "Method Area에 공유인데 왜 못 보지?" (두 개의 다른 층위)
가장 헷갈리는 지점이다. "static 변수는 Method Area에 있어 모든 스레드가 공유한다"(PART 2.2)고 배웠는데,
어떻게 한 스레드가 바꾼 걸 다른 스레드가 못 볼까? 답은 **서로 다른 두 층위를 구분**해야 풀린다.

- **층위 1 — JVM 논리적 메모리(Method Area / Heap / Stack)**: "변수가 논리적으로 어디 속하나"의 그림.
  `static running`은 **Method Area에 단 하나** 있고 모든 스레드가 같은 변수를 본다. 변수는 하나뿐 — 이건 맞다.
- **층위 2 — 하드웨어 메모리 계층(메인 메모리 RAM ↔ 코어별 캐시 ↔ 레지스터)**: "CPU가 실제 실행할 때
  값을 어디로 복사해 쓰나"의 그림. 코어는 느린 메인 메모리 대신 **자기 캐시/레지스터에 값을 복사해 와서** 일한다.

즉 **논리적으로는 변수가 하나(공유)지만, 물리적으로는 각 코어가 '복사본'을 들고 돈다.** 그 복사본들이
서로·원본과 어긋나는 것이 가시성 문제다. "공유한다"가 "항상 즉시 똑같이 동기화된다"를 뜻하지는 않는다.

```
층위1(논리): Method Area 의 running  = 단 하나(공유)
                         │  (실행 시 이 값이 결국 메인 메모리 RAM에 있음)
층위2(물리): [코어1(워커) 캐시/레지스터: running=true(옛 복사본)]
            [코어2(main)  캐시: running=false(방금 바꿈)]
                         └──── 메인 메모리(RAM): running ────┘
```
main(코어2)이 false로 바꿔도 워커(코어1)가 든 옛 복사본은 갱신 안 될 수 있어 → 워커는 계속 true를 본다.

> 헷갈림 정리 — **"캐시는 Stack별인가?" → 아니다.** Stack은 JVM '논리 영역'(스레드별, 층위1)이고,
> 캐시는 '하드웨어'(CPU **코어별**, 층위2)다. 레지스터도 하드웨어이며 스레드 실행 문맥별이다. 셋은
> 서로 다른 것 — 캐시는 Method Area/Stack 같은 JVM 논리 영역과 무관하다. "어디에 저장되나(논리)"와
> "실행 중 어디로 복사해 쓰나(물리)"를 분리해서 보자.
>
> 비유: Method Area의 running = 칠판에 붙은 공지 한 장(공유, 하나). 각 직원(코어)이 칠판을 매번 안 보고
> 수첩에 베껴두고 본다(캐시 복사본). main이 칠판을 고쳐도 워커가 자기 수첩만 보면 모른다. `volatile`은
> "수첩에 베끼지 말고 매번 칠판을 직접 보라"는 명령이다.

이제 그 "복사해 쓰기"가 구체적으로 왜 일어나는지 두 가지로 본다.

(1) **CPU는 메인 메모리를 직접 안 쓰고, 각 코어의 캐시를 쓴다.**
메인 메모리(RAM)는 CPU에 비해 느리다. 그래서 각 CPU 코어는 자기 옆에 빠른 **캐시(cache)**를 두고,
변수 값을 거기 복사해 와서 일한다. 문제는 **코어마다 캐시가 따로**라는 점이다.
```
   [코어1 캐시: running=true]      [코어2 캐시: running=true]
            │                              │
            └────── 메인 메모리(running) ──────┘
```
코어1이 자기 캐시의 running을 false로 바꿔도, 그게 메인 메모리와 코어2 캐시까지 **언제 반영될지 보장이
없다.** 코어2는 한참 동안 옛 값(true)만 볼 수 있다. → 이것이 가시성 문제의 뿌리.

(2) **컴파일러/CPU는 코드를 "최적화"하면서 재배치·캐싱한다.**
JIT 컴파일러는 `while(running){}`처럼 루프 안에서 running이 안 바뀌어 '보이면', 매번 메모리를 읽는 게
낭비라고 판단해 **값을 레지스터에 한 번만 읽어두고 재사용**한다(루프 밖으로 빼냄). 그러면 다른 스레드가
running을 바꿔도 이 스레드는 영원히 옛 값만 본다.

> 정리: 멀티스레드에서 **"내가 방금 쓴 값을, 다른 스레드가 즉시·정확히 본다"는 보장이 기본적으로 없다.**
> 이 보장이 깨질 때 생기는 버그가 아래 두 가지(가시성·원자성)다. 동기화 도구(volatile/synchronized/Atomic)는
> 사실 "이 보장을 다시 세워주는" 장치들이다(7.5).

동시성 버그는 크게 두 가지로 나뉜다. **둘은 다른 문제이고 해결 도구도 다르다.**

| 개념 | 개념의 정의(성질) | 그 성질이 깨질 때의 문제 | 대표 증상 |
|---|---|---|---|
| **가시성(Visibility)** | 한 스레드가 쓴 값이 다른 스레드에 보이는 성질 | 변경이 전파 안 돼 다른 스레드가 옛 값을 봄 | `running=false` 했는데 무한 루프 |
| **원자성(Atomicity)** | 연산이 쪼개지지 않고 한 번에 일어나는 성질 | 다단계 연산 도중 다른 스레드가 끼어들어(인터리빙) 갱신이 덮어써짐 | `count++`가 일부 손실 |

> 용어 정리: '가시성/원자성'은 원래 **연산이 갖춰야 할 좋은 성질(개념)**의 이름이다. 그 성질이 보장되지
> 않을 때 생기는 게 '가시성 문제 / 원자성 문제'다. 즉 "동시 수정이 서로 덮어씀"은 원자성의 *정의*가
> 아니라 **원자성이 깨졌을 때의 문제(증상)**다. 이 문서에서 "가시성/원자성 문제"라고 할 때는 '그 성질이
> 깨진 상태'를 가리킨다.

### 1-1. 가시성(Visibility) 문제 — "변경이 안 보인다"
**무엇**: 한 스레드가 공유 변수를 바꿨는데, 다른 스레드가 그 변경을 못 보고 **옛 값**을 계속 읽는 것.

**왜**: 위 토대 (1)(2) 때문이다. 변경한 값이 메인 메모리/다른 코어 캐시까지 전파될 보장이 없고, JIT가
값을 캐싱해버리면 아예 다시 안 읽는다.

**전형적 시나리오 — 종료 플래그**:
```java
static boolean running = true;          // volatile 아님

// 워커 스레드
while (running) { /* 일함 */ }           // running이 false로 바뀌길 기다림
System.out.println("멈춤");

// main 스레드
running = false;                         // "이제 그만!" 신호
```
의도는 "main이 running=false로 바꾸면 워커가 루프를 빠져나온다"이다. 그런데 워커가 running을 자기
캐시/레지스터에 캐싱해버리면, main의 변경을 **영원히 못 보고 무한 루프**를 돈다(프로그램이 안 끝남).

**해결**: `volatile boolean running`. `volatile`을 붙이면 "이 변수는 항상 메인 메모리에서 직접 읽고
쓰라(캐시에 가두지 말라)"는 뜻이 되어, 한 스레드의 변경이 다른 스레드에 즉시 보인다.

> ★ 헷갈리는 지점 — "가시성 문제는 항상 재현되나?" → 아니다. JVM·OS·CPU·최적화 상황에 따라
> 어떤 때는 변경이 (늦게라도) 보여 멈추기도 한다. 그래서 더 위험하다 — "내 PC에선 잘 되는데 서버에선
> 무한 루프"가 될 수 있다. "어쩌다 되는 것"에 기대지 말고 volatile/동기화로 **항상 보이게** 만들어야 한다.

### 1-2. 원자성(Atomicity) 문제 — "동시 수정이 덮어쓴다"
**원자적(atomic)** = "더 쪼갤 수 없이 한 번에" 일어나는 연산. 문제는 `count++`가 한 줄처럼 보여도
**원자적이지 않다**는 것이다. 실제로는 3단계로 쪼개진다.
```
count++  ≡  (1) 읽기: 메모리의 count를 읽어옴      (예: 100)
            (2) 증가: 읽은 값에 +1                  (101)
            (3) 쓰기: 결과를 메모리에 저장          (count=101)
```
단일 스레드면 문제없다. 하지만 두 스레드가 이 3단계를 **번갈아 끼어들면(인터리빙)** 갱신이 사라진다.
```
시간 →
스레드A: (1)읽기 100 ───────────── (2)+1=101 ── (3)쓰기 101
스레드B:        (1)읽기 100 ── (2)+1=101 ── (3)쓰기 101
결과: 둘 다 100을 읽고 둘 다 101을 씀 → 2번 증가했는데 count=101 (1번 유실!)
```
이렇게 한쪽의 증가가 다른 쪽에 덮여 사라지는 것을 **갱신 유실(lost update)**, 이런 타이밍 의존 버그가
생기는 상황을 **경쟁 상태(race condition)**라 한다. 그래서 N번 증가시켜도 결과가 N보다 작게 나온다.

**비유**: 통장 잔액 100. 두 사람이 동시에 "잔액 100 확인 → 100원 입금 → 200 기록"을 하면, 둘 다 100을
보고 시작했으므로 최종 잔액이 300이 아니라 200이 된다(한 명의 입금이 사라짐).

> ★ 헷갈리는 지점 — "그럼 `count = count + 1`은 원자적인가?" → 역시 아니다. `count++`의 다른 표기일
> 뿐, 똑같이 읽기-더하기-쓰기 3단계다. "한 줄로 썼다"와 "원자적이다"는 전혀 다른 얘기다.

### 1-3. ★ 둘은 별개의 문제 — volatile은 가시성만 해결한다 (가장 중요)
가장 흔한 오해가 "volatile 붙이면 다 안전하다"이다. **틀렸다.** 두 문제는 별개이고, volatile은 그중
가시성만 해결한다.

- `volatile boolean flag` → **가시성 해결**. 항상 메인 메모리에서 R/W하니 변경이 즉시 보인다(1-1 해결).
- `volatile int count; count++` → **원자성 미해결**. volatile은 (1)읽기와 (3)쓰기를 각각 메인 메모리와
  맞춰줄 뿐, "(1)읽기-(2)증가-(3)쓰기 3단계를 하나로 묶어주지는 못한다." 그래서 두 스레드가 그 사이에
  끼어들면 **여전히 유실**된다.

왜 그런가: volatile이 보장하는 건 "각 읽기/쓰기 한 번"의 최신성이지, "여러 단계를 통째로 보호"하는 게
아니다. 원자성은 "그 3단계 동안 아무도 끼어들지 못하게 막는" 것이 필요한데, 그건 락(synchronized)이나
CAS(Atomic)의 역할이다.

| 필요한 것 | 적합한 도구 |
|---|---|
| 가시성만 (예: 종료 플래그를 한 스레드가 쓰고 여럿이 읽음) | `volatile` |
| 원자성 (예: 여러 스레드가 같은 카운터를 증가) | `synchronized` 또는 `AtomicInteger`(CAS) |
| 둘 다 | `synchronized` / `Atomic` (가시성도 함께 보장됨) |

→ 이 구분이 7.5(synchronized / volatile / Atomic 비교)의 출발점이다.

### 실행 결과 — 가설과 실제 비교

**예시 1 (가시성)** — 가설 1.
- main이 `running=false`로 바꿔도 worker가 못 보고 **2초 뒤에도 살아있음(무한 루프)** → 가시성 문제 재현. ✅
- (worker는 데몬이라 main 종료 시 JVM이 정리. 이 문제는 JVM/CPU 의존이지만 JDK 21에서 안정 재현.)

**예시 2 (원자성)** — 가설 2.
- 2스레드 × 1,000,000 = 기대 2,000,000인데 실제 **1,061,403** (약 93만 유실). count++가 원자적이지 않음. ✅

**예시 3 (둘은 별개)** — 가설 3.

| | 결과 |
|---|---|
| (A) `volatile boolean flag` | worker가 변경을 보고 **즉시 종료** (가시성 해결) |
| (B) `volatile int count++` | 기대 2,000,000인데 **932,322** (원자성 미해결, 여전히 유실) |

→ volatile은 가시성만 해결, 원자성은 못 해결. 두 문제는 별개. ✅

### 세 예시를 관통하는 결론
동시성 버그는 **가시성**(변경이 안 보임 — 예시1)과 **원자성**(동시 수정이 덮어씀 — 예시2)의 두 갈래다.
둘은 별개이고, `volatile`은 가시성만 해결한다(예시3). 그래서 도구를 문제에 맞게 골라야 한다 —
가시성만 필요하면 volatile, 원자성(또는 둘 다)이 필요하면 synchronized/Atomic. 이 분해가 7.5
(synchronized/volatile/Atomic 비교)의 출발점이다.

## 3. 자기 점검

- **Q. 가시성과 원자성 문제의 차이는?**
  - 내 답: 가시성=한 스레드의 변경이 다른 스레드에 안 보임(캐시/최적화 탓). 원자성=count++ 같은
    다단계 연산이 동시에 진행돼 갱신이 유실됨. 둘은 별개 문제. (Example1 vs Example2)

- **Q. count++가 원자적이지 않은 이유는?**
  - 내 답: 읽기 → +1 → 쓰기 3단계라, 두 스레드가 같은 옛 값을 읽고 각자 +1해 쓰면 한 번만 반영된다. (Example2)

- **Q. volatile로 count++를 안전하게 만들 수 있나?**
  - 내 답: 없다. volatile은 가시성만 해결하고, 읽기-증가-쓰기 3단계를 묶지 못해 원자성은 미해결.
    synchronized나 AtomicInteger(CAS)가 필요하다(7.5). (Example3 B)

- **Q. 애초에 왜 한 스레드의 변경이 다른 스레드에 안 보일 수 있나?**
  - 내 답: 각 CPU 코어가 자기 캐시에 값을 들고 일하고(코어마다 캐시 따로), JIT가 값을 레지스터에
    캐싱해 재사용하기 때문. "내가 쓴 값을 남이 즉시 본다"는 보장이 기본적으로 없다. (1-0 토대)

- **Q. 가시성 버그가 "내 PC에선 잘 되는데 서버에선 안 됨"이 되는 이유는?**
  - 내 답: 가시성 문제는 JVM·OS·CPU·최적화 상황에 따라 재현될 때도, 안 될 때도 있어서. "어쩌다
    되는 것"에 의존하면 환경이 바뀌면 터진다 → volatile로 항상 보이게 해야 한다. (1-1)

- **Q. `count = count + 1`은 `count++`와 달리 원자적인가?**
  - 내 답: 아니다. 표기만 다를 뿐 똑같이 읽기-더하기-쓰기 3단계라 동시 수정 시 유실된다. (1-2)

- **Q. static 변수는 Method Area에 '공유'인데 왜 다른 스레드가 변경을 못 보나?**
  - 내 답: 두 층위가 다르다. 논리적으로 변수는 Method Area에 하나(공유)지만, 실행 시 각 CPU 코어가
    그 값을 자기 캐시/레지스터에 '복사'해 쓰기 때문에 복사본이 어긋날 수 있다. 공유 ≠ 자동 동기화. (1-0)

- **Q. CPU 캐시는 스레드 Stack별로 있나?**
  - 내 답: 아니다. 캐시는 하드웨어로 CPU '코어별'이고, Stack은 JVM 논리 영역으로 '스레드별'이다.
    서로 다른 층위라 무관하다. (1-0)
