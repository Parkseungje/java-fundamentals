## 1. 학습 내용 — 프로세스, 스레드, 메모리, 컨텍스트 스위칭

### 멀티태스킹 vs 멀티프로세싱
- **멀티태스킹**: 코어 1개가 시분할로 여러 작업을 빠르게 번갈아 → 동시처럼 보임.
- **멀티프로세싱**: 여러 코어가 진짜 동시에 실행.
- 둘은 함께 쓰인다(예: 12코어가 각자 시분할로 여러 스레드를 번갈아).

### 프로세스와 스레드
- **프로세스**: 실행 중인 프로그램(예: 이 JVM). 독립된 메모리를 갖고, **1개 이상의 스레드**를 포함한다.
- **스레드**: 프로세스 안에서 코드를 실행하는 **흐름**. `main()`도 'main 스레드'라는 하나의 흐름이다.
- 한 프로세스 안 여러 스레드는 동시에(또는 번갈아) 실행되며, 그래서 **출력 순서가 매번 달라진다**(동시성의 특징).

### 메모리 공유 — 무엇을 공유하고 무엇이 따로인가 (★ 핵심)
스레드는 같은 프로세스 안에 있어 메모리를 **일부 공유, 일부 분리**한다(2.2/2.3).

| 영역 | 공유 여부 |
|---|---|
| 코드 / 데이터 / Heap / Method Area | **공유** (모든 스레드가 같이 봄) |
| Stack (지역 변수) | **스레드별로 따로** |

### 스레드 안전성 = "공유되는가?"
그래서 "이 변수가 스레드 안전한가?"는 사실상 **"이 변수가 공유되는가?"** 와 같은 질문이다.

| 변수 | 위치 | 안전성 |
|---|---|---|
| 지역 변수 (Local) | Stack(스레드별) | **안전**(공유 X) |
| 인스턴스 변수 (Instance) | Heap | 공유 → 동기화 필요 |
| static 변수 | Method Area | 공유 → 동기화 필요 |

### 컨텍스트 스위칭
스레드를 전환할 때 CPU는 현재 스레드의 레지스터 상태를 백업하고 다음 것을 복원하며, 캐시도
무효화된다 — 이 비용이 **컨텍스트 스위칭**이다. 코어 수보다 스레드가 너무 많으면 CPU가 '실제 작업'보다
'전환'에 시간을 더 써서 오히려 느려진다. **CPU 바운드 작업의 적정 스레드 수 ≈ 코어 수.**
(반대로 I/O 바운드는 대기가 많아 코어 수보다 많은 스레드가 유리할 수 있다 — 6.2 NIO와 대비.)

## 2. 실습으로 확인하기

> - **가설 1**: 한 프로세스 안 여러 스레드가 동시에 실행되어 출력이 뒤섞인다.
> - **가설 2**: 지역 변수(Stack)는 스레드별이라 안전, 공유 static 변수는 동기화 없으면 유실된다.
> - **가설 3**: CPU 바운드 작업은 스레드를 코어 수 이상 늘려도 안 빨라지고, 과하면 스위칭으로 느려진다.

### 예시 3개 — 각 예시가 답하는 질문

| 예시 | 답하는 질문 | 시나리오 |
|---|---|---|
| `Example1_ProcessAndThreads` | 동시 실행이란? | main + worker 2개 인터리빙 |
| `Example2_SharedVsLocal` | 무엇이 안전한가? | 지역 변수 vs 공유 static |
| `Example3_ContextSwitching` | 스레드 많을수록 빠른가? | 코어 수 vs 50배 스레드 |

### 실행
아래 명령은 모두 **프로젝트 루트(`C:\develop\study\java-fundamentals`)에서 실행**한다.
(docs 폴더 등 다른 위치에서 실행하면 `build/...` 상대경로를 못 찾아 에러가 난다. 그럴 땐 `cd ..`로 루트로 이동)

```bash
./gradlew compileJava
java -cp build/classes/java/main com.study.part07_concurrency.s01_process_thread.Example1_ProcessAndThreads
java -cp build/classes/java/main com.study.part07_concurrency.s01_process_thread.Example2_SharedVsLocal
java -cp build/classes/java/main com.study.part07_concurrency.s01_process_thread.Example3_ContextSwitching
```

### 실행 결과 — 가설과 실제 비교

**예시 1 (동시 실행)** — 가설 1.
- PID와 main 스레드 출력 후, main/worker-1/worker-2의 출력이 **뒤섞여** 나온다(순서 매번 다름). ✅
- → 한 프로세스(JVM) 안에서 세 스레드가 동시에 흐른다는 증거.

**예시 2 (공유 vs 지역)** — 가설 2.

| 방식 | 결과 |
|---|---|
| 지역 변수 | 4스레드 모두 정확히 100000 (독립, 안전) |
| 공유 static | 기대 400000인데 **114541** (유실 — 경쟁 상태) |

→ 지역 변수는 스레드별이라 안전, 공유 static은 동기화 없으면 유실. "안전 = 공유 안 됨". ✅

**예시 3 (컨텍스트 스위칭)** — 가설 3. (12코어, JIT 워밍업 후 실측)

| 스레드 수 | 시간 |
|---|---|
| 12개 (코어 수) | 343 ms (코어 병렬로 꽉 채움) |
| 600개 (코어의 50배) | 380 ms (더 안 빨라짐 + 스위칭 오버헤드) |

→ CPU 바운드는 코어 수 이상 스레드를 만들어도 안 빨라지고, 과하면 스위칭 비용으로 오히려 느려진다. ✅
(※ 측정 전 JIT 워밍업을 했다 — 안 하면 먼저 잰 쪽이 미완 컴파일이라 손해 보여 결과가 왜곡된다.)

### 세 예시를 관통하는 결론
한 프로세스 안 여러 스레드는 메모리를 공유하며 동시에 흐른다(예시1). 그래서 무엇을 공유하느냐가
스레드 안전성을 가른다 — 지역 변수(Stack)는 안전하고 공유 변수(Heap/Method Area)는 동기화가
필요하다(예시2). 그리고 스레드는 공짜가 아니라 컨텍스트 스위칭 비용이 있어, CPU 바운드에선 코어 수가
적정선이다(예시3). 이 "공유 변수의 위험"이 7.4(가시성·원자성)와 7.5(synchronized/volatile/Atomic)로 이어진다.

## 3. 자기 점검

- **Q. 스레드 안전성을 한 질문으로 줄이면?**
  - 내 답: "이 변수가 공유되는가?" 지역 변수(Stack)는 스레드별이라 안전, 인스턴스/static 변수는
    공유되어 동기화가 필요하다. (Example2)

- **Q. CPU 바운드 작업의 적정 스레드 수는? 왜?**
  - 내 답: 대략 코어 수. 그 이상은 병렬도가 안 늘고 컨텍스트 스위칭 비용만 추가되기 때문. (Example3)

- **Q. (구분) I/O 바운드 작업은 왜 코어 수보다 많은 스레드가 유리할 수 있나?**
  - I/O 대기 중인 스레드는 CPU를 안 쓰므로, 그동안 다른 스레드가 CPU를 쓸 수 있다. 그래서 대기가
    많은 작업은 코어보다 많은 스레드로 처리량을 높일 수 있다(6.2 NIO/Non-blocking과 연결해 정리).
