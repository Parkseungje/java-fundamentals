## 1. 학습 내용 — 프로세스, 스레드, 메모리, 컨텍스트 스위칭

### ★ 먼저 용어 정리 — 비슷한 4개를 구분하자
이름이 비슷해 헷갈리는 용어들이다. 층위가 다르다.

| 용어 | 뜻 | 층위 |
|---|---|---|
| 멀티태스킹(multitasking) | OS가 여러 작업을 시분할로 번갈아 실행 | 실행 방식 |
| 멀티프로세싱(multiprocessing) | 여러 CPU 코어가 진짜 동시에 실행 | 하드웨어 |
| 멀티프로세서(multiprocessor) | CPU 코어가 여러 개인 것 | 하드웨어 |
| **멀티프로세스(multi-process)** | **한 응용프로그램을 여러 프로세스로 구성**(부모/자식, 독립 메모리, IPC) | **소프트웨어 구조** |
| **멀티스레드(multi-thread)** | **한 프로세스 안에 여러 스레드로 구성** | **소프트웨어 구조** |

- **멀티태스킹 vs 멀티프로세싱**: 코어 1개가 번갈아(태스킹) vs 여러 코어가 동시에(프로세싱). 둘은
  함께 쓰인다(예: 12코어가 각자 시분할로 여러 스레드를 번갈아).
- **멀티프로세스 vs 멀티스레드**: "한 앱을 어떻게 구성하느냐"는 소프트웨어 처리 방식의 선택이다(아래에서
  장단점을 비교한다). 이 7.1의 핵심 주제다. 위의 '멀티프로세싱(하드웨어)'과 헷갈리지 말 것.

### 멀티프로세스 vs 멀티스레드 — 한 앱을 구성하는 두 방식 (★ 7.1 핵심)
멀티프로세스/멀티스레드는 "프로그램을 여러 개 띄우는 것"이 아니라, **하나의 응용프로그램을 어떤 단위로
나눠 동시 처리하느냐**의 선택이다.

- **멀티프로세스**: 한 앱이 여러 프로세스로 동작. 부모 프로세스가 자식 프로세스를 만들고(각자 고유 PID),
  **각 프로세스는 독립된 메모리 공간**을 가진다. 예: 웹 브라우저의 탭/창(탭마다 별도 프로세스라, 한 탭이
  멈춰도 다른 탭은 멀쩡). 자원 공유가 필요하면 **IPC**(파이프·소켓·메시지 큐 등)를 써야 한다.
- **멀티스레드**: 한 프로세스 안에서 여러 스레드로 동작. 스레드들은 **코드/데이터/Heap을 공유**하고
  Stack만 따로 가진다. 예: 브라우저 한 탭 안에서 네트워크·렌더링·이벤트 처리를 동시에.

#### 멀티프로세스의 장단점
- 장점
  1. **안정성**: 프로세스가 독립 메모리라, 한 프로세스가 죽어도 다른 프로세스는 안 죽는다(브라우저 한
     탭이 먹통이어도 다른 탭은 멀쩡).
  2. **병렬성**: 여러 코어에 프로세스를 분산해 병렬 처리(단, 이건 멀티스레드도 누리는 이점이고, 보통
     스레드가 더 효율적이라 성능 목적의 멀티프로세스는 드물다).
  3. **확장성**: 프로세스가 독립적이라 서버 프로세스를 여러 개 띄워 부하 분산(클러스터)하기 쉽다.
- 단점
  1. **컨텍스트 스위칭 비용 큼**: 프로세스 전환은 메모리 맵·CPU 캐시까지 갈아끼워 무겁다(스레드 전환보다 비쌈).
  2. **자원 공유 비효율**: 메모리를 따로 쓰니 사용량이 늘고, 공유하려면 복잡하고 느린 **IPC**가 필요하다.

#### 멀티스레드의 장단점
- 장점
  1. **가벼움**: 스레드는 프로세스 안에서 생기고 자원을 공유해 생성/종료가 빠르고 메모리가 작다.
  2. **자원 효율**: Heap 등을 공유하므로 IPC 없이 데이터를 주고받는다.
  3. **컨텍스트 스위칭 저렴**: 공유 자원은 그대로 두고 스레드 정보(스택·레지스터)만 교체 → 프로세스
     전환보다 훨씬 가볍다.
  4. **응답성**: 위 이유들로 멀티프로세스보다 응답이 빠르다(웹 서버가 요청마다 프로세스 대신 스레드를 쓰는 이유).
- 단점
  1. **안정성 취약**: 한 스레드의 오류가 프로세스 전체를 죽일 수 있다(메모리를 공유하므로). 예외 처리·
     스레드풀로 완화한다.
  2. **동기화 비용**: 공유 자원을 보호하려 동기화(뮤텍스/세마포어 등)하면 병목이 생겨 성능이 떨어질 수 있다(7.5).
  3. **데드락**: 여러 스레드가 서로의 락을 기다려 멈출 수 있다(7.6). (단 데드락은 IPC를 쓰는 멀티프로세스도 겪는다.)
  4. **그래도 컨텍스트 스위칭 비용**: 스레드 전환도 공짜는 아니라, 스레드가 너무 많으면 느려진다(아래 예시3).
  5. **디버깅 어려움**: 실행 순서가 비결정적이라 추적이 어렵다.

> 결론: 대부분의 경우 **멀티스레드가 더 가볍고 효율적**이라, 성능을 위해선 멀티스레드를 쓴다. 멀티프로세스는
> '안정성·격리'가 중요할 때(브라우저 탭, 서버 프로세스 분리) 선택한다. 이 7.1의 코드 예시들은 자바의
> **멀티스레드**에 초점을 둔다(아래).

#### 메모리 격리 — 멀티스레드는 Stack만, 멀티프로세스는 전부
"무엇이 격리되느냐"가 둘의 본질적 차이다.

- **멀티스레드(한 프로세스 안)**: 같은 프로세스의 메모리를 공유하고 **Stack만 스레드별**로 따로.

  | 영역 | 공유? |
  |---|---|
  | 코드(Text) / Data / Heap / Method Area | ✅ 공유 |
  | Stack (+ PC, 레지스터) | ❌ 스레드별 |

- **멀티프로세스(별도 프로세스)**: 프로세스마다 **완전히 독립된 주소 공간**이라 **전부 격리**.

  | 영역 | 격리? |
  |---|---|
  | 코드(Text) / Data / Heap / Method Area / Stack | ✅ 전부 따로 |

"공유한다"의 기준은 **프로세스**다. 스레드는 *같은 프로세스 안*이라 그 프로세스의 메모리(Method Area·
Data·Heap)를 같이 쓰고, 다른 프로세스는 *애초에 다른 메모리 공간*이라 같은 이름의 영역이라도 물리적으로
별개다. 그래서 JVM 2개를 띄우면 **각 JVM이 자기만의 Method Area·Heap·Data를 따로** 갖고 서로 못 본다.
프로세스끼리 데이터를 나누려면 메모리를 직접 공유 못 하고 **IPC**(파이프·소켓·메시지 큐)가 필요하다.
(참고: Method Area는 JVM 용어이고, OS 관점에선 그냥 "그 프로세스 메모리의 일부"다.)

비유: 스레드 = 한 집(프로세스) 안 식구들 → 거실·주방(공유 메모리)은 같이, 각자 방(Stack)만 따로.
멀티프로세스 = 아예 다른 집들 → 전부 별개라 뭔가 주고받으려면 택배(IPC)를 보내야 한다.

#### 실전 — 작업관리자에 '프로세스 여러 개'로 보이는 앱들 (왜?)
Windows 작업관리자에서 한 프로그램이 하위 프로세스 여러 개로 펼쳐지는 것들(예: Chrome ▸ 프로세스 10개)이
있다. 대표적으로 **웹 브라우저**(Chrome/Edge/Firefox — 탭·확장·GPU·렌더러가 별도 프로세스), **Electron 앱**
(VS Code·Slack·Discord·Notion — 메인 + 렌더러들), 일부 메신저·게임 런처·안티바이러스다.

왜 멀티프로세스로 만드나:
1. **안정성(장애 격리)** — 메모리가 완전 격리라 하나가 죽어도 나머지는 멀쩡(크롬 한 탭이 멈춰도 다른 탭은
   계속). 멀티스레드였다면 한 탭 크래시가 전체를 죽였을 것(멀티스레드 단점을 멀티프로세스로 회피).
2. **보안(샌드박싱)** — 탭/렌더러를 권한 제한된 별도 프로세스에 가둔다. 악성 페이지가 렌더러를 장악해도
   프로세스 경계가 벽이 되어 내 파일·다른 탭으로 못 넘어온다.
3. **위험·취약 컴포넌트 분리** — GPU·네트워크·플러그인·오디오처럼 잘 죽거나 위험한 부분을 따로 떼어 둔다.
4. **(Electron) 구조적 강제** — Chromium 기반이라 '메인 1개 + 창마다 렌더러'를 그대로 물려받는다.

공통점: **"한 부분의 사고가 전체로 번지면 치명적인" 앱**들이다(임의의 웹코드/플러그인 실행). 그래서
성능을 조금 손해 보더라도(메모리 더 씀 — "크롬이 메모리 많이 먹는다"의 원인) 안정성·보안을 위해 격리한다.

#### 작업관리자에서 PID 없는 맨 윗줄은 부모 프로세스인가? (아니오 — 그룹 헤더)
작업관리자 '프로세스' 탭에서 `Google Chrome(27)`처럼 **PID 없이 맨 위에 뜨는 줄은 부모 프로세스가
아니라, 같은 프로그램의 프로세스 여러 개를 묶어 접어놓은 '그룹 머리글'** 이다.
- `(27)` = 이 그룹에 프로세스가 27개 있다는 뜻.
- PID가 없는 이유 = **그 줄 자체는 프로세스가 아니라서**(화면 표시용 묶음일 뿐).
- 왼쪽 펼침 화살표를 펼치면 그제야 PID 가진 진짜 프로세스 27개가 나온다. 맨 윗줄의 메모리 합계(예:
  1,941MB)도 **27개 프로세스의 합**이다.

그럼 진짜 부모(브라우저 프로세스)는? **PID 달린 27개 중 하나**다(보통 조율·UI를 맡아 메모리를 좀 더
쓰는 편). 나머지는 렌더러(탭)·GPU·유틸리티·확장(extension-host)·네이티브 호스트 등 자식들이다. 다만
이름이 다 "Google Chrome"이라 '프로세스' 탭만으론 누가 부모인지 정확히 구분되지 않는다.

부모-자식 관계를 정확히 보려면:
1. 작업관리자 → **'세부 정보(Details)' 탭** → 컬럼 우클릭 → **'부모 PID(PPID)'** 추가.
2. 더 명확히는 **Process Explorer**(MS 공식 무료 툴) — 부모-자식을 **트리 구조**로 보여준다.

#### 그럼 부모 프로세스가 죽으면 자식도 다 영향받지 않나? (단일 장애점 의문)
맞는 우려다 — **부모 프로세스는 전체의 단일 장애점이 될 수 있다.** 그래도 이 설계가 효과적인 이유는
**"위험을 자식으로 몰아넣기" 때문**이다.

먼저 오해 정정: 부모-자식은 '생성·관리' 관계지 '메모리 공유' 관계가 아니다. 자식도 독립 메모리라 부모가
죽어도 자식 메모리가 손상되거나 자동으로 같이 터지진 않는다(OS 관점에선 자식이 '고아 프로세스'로 살 수도).

핵심은 역할 분담이다:

| | 부모 프로세스 | 자식 프로세스 |
|---|---|---|
| 역할 | 조율·UI·창 관리 (단순·신뢰됨) | 실제 위험한 일 (웹 렌더링, 임의 코드 실행) |
| 잘 죽나? | 거의 안 죽음 (하는 일이 단순·안전) | 잘 죽음 (악성 웹·버그·플러그인) |
| 죽으면? | 앱 전체 종료 (드묾) | 그 자식만 격리되어 죽음 (흔함) |

즉 "잘 터지는 위험한 일"을 전부 자식에게 떠넘기고 부모는 가볍고 안전한 조율만 한다. 그래서 **자주
일어나는 사고(자식 크래시)는 격리되어 영향 없고, 부모 크래시는 부모가 위험을 안 만지니 거의 안 난다.**
확률로 보면: 위험 작업 10개를 한 프로세스에 다 넣으면 사고 시 100% 전체 다운, 자식 10개로 쪼개면 사고
대부분이 자식 1개만 죽이고 부모가 죽는 드문 경우만 전체 다운 → **전체가 죽을 확률을 확 낮춘 것**이다.

부모가 죽으면 앱 전체가 사실상 못 쓰게 되므로(조율자 소멸), 보통 앱은 부모 종료 시 자식들도 정리하도록
설계한다. 하지만 이건 드문 실패 모드이고, 멀티스레드라면 '자식 하나의 사고 = 곧 전체 다운'인데
멀티프로세스는 그 흔한 사고를 격리한 것이라 트레이드오프상 이득이다.

> 한 줄 정리: 멀티프로세스는 "절대 안 죽는 시스템"이 아니라, **자주 터지는 위험한 부분(자식)을 격리해
> '전체가 죽는 사고'를 희귀한 경우(부모 사망)로 줄이는** 설계다. 부모는 단일 장애점이 맞지만, 위험을 안
> 만지게 해서 그 확률을 최소화한 것이다.

### 프로세스와 스레드
- **프로세스**: 실행 중인 프로그램(예: 이 JVM). 독립된 메모리를 갖고, **1개 이상의 스레드**를 포함한다.
- **스레드**: 프로세스 안에서 코드를 실행하는 **흐름**. `main()`도 'main 스레드'라는 하나의 흐름이다.
- 한 프로세스 안 여러 스레드는 동시에(또는 번갈아) 실행되며, 그래서 **출력 순서가 매번 달라진다**(동시성의 특징).

### 메모리 공유 — 무엇을 공유하고 무엇이 따로인가 (★ 핵심)
스레드는 같은 프로세스 안에 있어 메모리를 **일부 공유, 일부 분리**한다(2.2/2.3).

| 영역 | 공유 여부 |
|---|---|
| 코드 / 데이터 / Heap / Method Area | **공유** (모든 스레드가 같이 봄) |
| Stack (지역 변수) | **스레드별로 따로** |

### 스레드 안전성 = "공유되는가?"
그래서 "이 변수가 스레드 안전한가?"는 사실상 **"이 변수가 공유되는가?"** 와 같은 질문이다.

| 변수 | 위치 | 안전성 |
|---|---|---|
| 지역 변수 (Local) | Stack(스레드별) | **안전**(공유 X) |
| 인스턴스 변수 (Instance) | Heap | 공유 → 동기화 필요 |
| static 변수 | Method Area | 공유 → 동기화 필요 |

### 컨텍스트 스위칭
스레드를 전환할 때 CPU는 현재 스레드의 레지스터 상태를 백업하고 다음 것을 복원하며, 캐시도
무효화된다 — 이 비용이 **컨텍스트 스위칭**이다. 코어 수보다 스레드가 너무 많으면 CPU가 '실제 작업'보다
'전환'에 시간을 더 써서 오히려 느려진다. **CPU 바운드 작업의 적정 스레드 수 ≈ 코어 수.**
(반대로 I/O 바운드는 대기가 많아 코어 수보다 많은 스레드가 유리할 수 있다 — 6.2 NIO와 대비.)

## 2. 실습으로 확인하기

> - **가설 1**: 한 프로세스 안 여러 스레드가 동시에 실행되어 출력이 뒤섞인다.
> - **가설 2**: 지역 변수(Stack)는 스레드별이라 안전, 공유 static 변수는 동기화 없으면 유실된다.
> - **가설 3**: CPU 바운드 작업은 스레드를 코어 수 이상 늘려도 안 빨라지고, 과하면 스위칭으로 느려진다.

### 예시 3개 — 각 예시가 답하는 질문

| 예시 | 답하는 질문 | 시나리오 |
|---|---|---|
| `Example1_ProcessAndThreads` | 동시 실행이란? | main + worker 2개 인터리빙 |
| `Example2_SharedVsLocal` | 무엇이 안전한가? | 지역 변수 vs 공유 static |
| `Example3_ContextSwitching` | 스레드 많을수록 빠른가? | 코어 수 vs 50배 스레드 |

### 실행
아래 명령은 모두 **프로젝트 루트(`C:\develop\study\java-fundamentals`)에서 실행**한다.
(docs 폴더 등 다른 위치에서 실행하면 `build/...` 상대경로를 못 찾아 에러가 난다. 그럴 땐 `cd ..`로 루트로 이동)

```bash
./gradlew compileJava
java -cp build/classes/java/main com.study.part07_concurrency.s01_process_thread.Example1_ProcessAndThreads
java -cp build/classes/java/main com.study.part07_concurrency.s01_process_thread.Example2_SharedVsLocal
java -cp build/classes/java/main com.study.part07_concurrency.s01_process_thread.Example3_ContextSwitching
```

### 실행 결과 — 가설과 실제 비교

**예시 1 (동시 실행)** — 가설 1.
- PID와 main 스레드 출력 후, main/worker-1/worker-2의 출력이 **뒤섞여** 나온다(순서 매번 다름). ✅
- → 한 프로세스(JVM) 안에서 세 스레드가 동시에 흐른다는 증거.

**예시 2 (공유 vs 지역)** — 가설 2.

| 방식 | 결과 |
|---|---|
| 지역 변수 | 4스레드 모두 정확히 100000 (독립, 안전) |
| 공유 static | 기대 400000인데 **114541** (유실 — 경쟁 상태) |

→ 지역 변수는 스레드별이라 안전, 공유 static은 동기화 없으면 유실. "안전 = 공유 안 됨". ✅

**예시 3 (컨텍스트 스위칭)** — 가설 3. (12코어, JIT 워밍업 후 실측)

| 스레드 수 | 시간 |
|---|---|
| 12개 (코어 수) | 343 ms (코어 병렬로 꽉 채움) |
| 600개 (코어의 50배) | 380 ms (더 안 빨라짐 + 스위칭 오버헤드) |

→ CPU 바운드는 코어 수 이상 스레드를 만들어도 안 빨라지고, 과하면 스위칭 비용으로 오히려 느려진다. ✅
(※ 측정 전 JIT 워밍업을 했다 — 안 하면 먼저 잰 쪽이 미완 컴파일이라 손해 보여 결과가 왜곡된다.)

### 세 예시를 관통하는 결론
한 프로세스 안 여러 스레드는 메모리를 공유하며 동시에 흐른다(예시1). 그래서 무엇을 공유하느냐가
스레드 안전성을 가른다 — 지역 변수(Stack)는 안전하고 공유 변수(Heap/Method Area)는 동기화가
필요하다(예시2). 그리고 스레드는 공짜가 아니라 컨텍스트 스위칭 비용이 있어, CPU 바운드에선 코어 수가
적정선이다(예시3). 이 "공유 변수의 위험"이 7.4(가시성·원자성)와 7.5(synchronized/volatile/Atomic)로 이어진다.

## 3. 자기 점검

- **Q. 스레드 안전성을 한 질문으로 줄이면?**
  - 내 답: "이 변수가 공유되는가?" 지역 변수(Stack)는 스레드별이라 안전, 인스턴스/static 변수는
    공유되어 동기화가 필요하다. (Example2)

- **Q. CPU 바운드 작업의 적정 스레드 수는? 왜?**
  - 내 답: 대략 코어 수. 그 이상은 병렬도가 안 늘고 컨텍스트 스위칭 비용만 추가되기 때문. (Example3)

- **Q. (구분) I/O 바운드 작업은 왜 코어 수보다 많은 스레드가 유리할 수 있나?**
  - I/O 대기 중인 스레드는 CPU를 안 쓰므로, 그동안 다른 스레드가 CPU를 쓸 수 있다. 그래서 대기가
    많은 작업은 코어보다 많은 스레드로 처리량을 높일 수 있다(6.2 NIO/Non-blocking과 연결해 정리).
