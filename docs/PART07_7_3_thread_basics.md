## 0. 들어가기 전에 — 스레드가 대체 뭔가 (기초부터)

용어가 처음이면 7.1을 떠올리며 다음 그림부터 잡자.

- **프로그램**: 디스크에 저장된 코드 파일(예: 우리가 만든 `.class`).
- **프로세스(process)**: 그 프로그램이 실행되어 메모리에 올라간 '실행 중인 인스턴스'. 우리가 `java ...`로
  자바를 실행하면 **JVM이라는 프로세스 하나**가 뜬다.
- **스레드(thread)**: 그 프로세스 '안에서 코드를 실제로 한 줄씩 실행해 나가는 흐름(실행 주체)'. 한
  프로세스 안에 스레드가 여러 개 있을 수 있고, 그들이 동시에(또는 번갈아) 코드를 실행한다.

> ★ 처음 가장 헷갈리는 사실 — `main()`도 '스레드'가 실행한다.
> 자바 프로그램을 켜면 JVM이 **'main'이라는 이름의 스레드**를 하나 자동으로 만들고, 그 스레드가
> `main()` 메서드를 실행한다. 즉 우리가 평소 짜는 평범한 코드도 사실은 "main 스레드 한 개가 위에서
> 아래로 실행하는 것"이다. 멀티스레드란 여기에 **스레드를 더 만들어** 여러 흐름이 동시에 코드를
> 실행하게 하는 것이다. 그래서 이 단원의 주제는 "흐름(스레드)을 더 만들고(생성), 상태를 살피고,
> 기다리고(join), 정리하는" 법이다. (`Thread.currentThread().getName()`을 찍어보면 보통 "main"이 나온다.)

이 단원에서 새로 만드는 스레드를 편의상 'worker(일꾼) 스레드'라고 부른다. 정리하면 무대에는 항상 **main
스레드**가 있고, 우리가 거기에 **worker 스레드**를 추가로 올려 동시에 일을 시키는 그림이다.

## 1. 학습 내용 — 스레드 생애와 다루는 법

### 1-1. 스레드의 일생 — 상태(state)와 그 전이
스레드는 태어나서 죽을 때까지 여러 '상태'를 거친다. 자바는 이 상태를 `Thread.State` 열거형으로 정의하고,
`thread.getState()`로 지금 어떤 상태인지 확인할 수 있다. 사람의 하루(잠자기 전 → 일하는 중 → 잠깐 쉼 →
하루 끝)에 빗대면 이해가 쉽다.

- **NEW**: 스레드 객체를 `new`로 만들었지만 **아직 `start()`를 안 부른** 상태. "고용은 했지만 아직 출근 전".
- **RUNNABLE**: `start()` 후 **실행 중이거나, 곧 실행되려고 CPU를 기다리는** 상태. "일하는 중(또는 차례 대기)".
  (자바는 'CPU에서 실제로 도는 중'과 'CPU 차례를 기다리는 중'을 굳이 나누지 않고 둘 다 RUNNABLE로 본다.)
- **BLOCKED**: 다른 스레드가 쥐고 있는 **락(lock, 자물쇠)** 이 풀리기를 기다리는 상태. (락은 7.5에서 자세히.
  지금은 "공유 자원에 한 번에 하나만 들어가게 하는 출입증"이라고만 알아두자.) "화장실 문 잠겨서 밖에서 대기".
- **WAITING**: `wait()`이나 인자 없는 `join()`처럼 **기한 없이 무기한** 기다리는 상태. "누가 깨워줄 때까지 잠".
- **TIMED_WAITING**: `sleep(ms)`, `wait(ms)`, `join(ms)`처럼 **정해진 시간 동안만** 기다리는 상태. "알람 맞춰 놓고 잠".
- **TERMINATED**: `run()`이 끝나 **실행이 완전히 종료된** 상태. "퇴근 완료". (한 번 끝난 스레드는 다시 못 켠다.)

전형적 흐름:
```
NEW  --start()-->  RUNNABLE  <-->  (BLOCKED / WAITING / TIMED_WAITING)  -->  TERMINATED
(생성)            (실행/대기)        (락·통지·시간 때문에 잠시 멈춤)         (종료)
```
RUNNABLE과 대기 상태들 사이를 여러 번 오갈 수 있다(일하다 잠깐 멈췄다 다시 일하고...). 그러다 `run()`이
끝나면 TERMINATED로 가서 끝난다.

> ★ 헷갈리는 지점 — `Thread.sleep()`은 '누가' 자나? → **그 줄을 실행 중인 스레드 자신**이 잔다.
> `Thread.sleep(...)`은 `Thread` 클래스의 **static 메서드**다. static이라 "특정 스레드 객체에게
> 자라고 명령"하는 게 아니라, **"지금 이 코드를 실행하고 있는 현재 스레드"를 재운다.** 코드 어디에도
> "main"이라고 안 적혀 있어도, main 메서드 안에서 이 줄을 부르면 그 줄을 실행하는 게 main이므로 **main이**
> 잔다. worker의 `run()`/람다 안에서 부르면 **worker가** 잔다. 같은 `Thread.sleep` 문장이라도 **누가
> 실행하느냐**로 자는 주체가 갈린다.
> - 그래서 main에서 `Thread.sleep`을 불러도 다른 worker 스레드는 **안 멈춘다**(각자 따로 흐른다).
> - 비유: `Thread.sleep()`은 "나 좀 잘게"라는 혼잣말이다. 말한 사람(현재 스레드)이 자는 것이지 옆 사람을 재우는 명령이 아니다.
> - (7.2에서 본 "제어권은 현재 스레드 기준"과 같은 맥락이다.)

### 1-2. 스레드 만드는 두 방법 — Thread 상속 vs Runnable 구현
새 스레드에게 "무슨 일을 시킬지(=`run()`에 담길 코드)"를 정하는 방법이 두 가지다.

(1) **Thread 클래스를 상속**해서 `run()`을 오버라이드:
```java
class MyThread extends Thread {       // Thread를 물려받아
    @Override public void run() { ... }  // 할 일을 Thread '안'에 직접 넣음
}
Thread t = new MyThread();            // 이 객체 = '스레드'이자 '할 일'
```

(2) **Runnable을 구현**(보통 람다)해서 Thread에 건네주기 — 권장:
```java
Runnable task = () -> { ... };        // 할 일을 Thread '밖'에 따로 만듦
Thread t = new Thread(task);          // Thread에 "이 일 해" 하고 작업을 주입
```

> 용어: **Runnable**은 메서드가 `run()` 하나뿐인 **함수형 인터페이스**(추상 메서드 1개짜리 인터페이스 —
> 5.4 참고)다. 추상 메서드가 하나라서 **람다(`() -> {...}`)로 간단히 만들 수 있다.** "이 스레드가 할 일"을
> 담는 상자라고 보면 된다.

#### 핵심 차이 — "할 일(run의 내용)을 어디에 두느냐"
둘 다 `start()`하면 결국 새 스레드가 `run()`을 실행한다(결과는 비슷). 진짜 차이는 **할 일을 어디에 두느냐**다.
- **Thread 상속**: 할 일을 Thread 클래스 **'안'에** 직접 넣는다 → 객체 하나가 '스레드이자 할 일'을 겸한다.
- **Runnable 주입**: 할 일을 Thread **'밖'에** 따로 두고 건네준다 → Thread(실행 수단)와 Runnable(할 일)이 분리된다.

| | Thread 상속 (`new MyThread()`) | Runnable 주입 (`new Thread(task)`) |
|---|---|---|
| 할 일 위치 | Thread **안**(run 오버라이드) | Thread **밖**(별도 Runnable) |
| 구성 | 스레드+할 일 한 몸 | 스레드 / 할 일 분리(2개) |
| 다른 클래스 상속 | ❌ (단일 상속을 Thread에 써버림) | ✅ (Runnable은 인터페이스) |
| 작업 재사용 | ❌ | ✅ 같은 task를 여러 Thread·ExecutorService(7.8)에 넘김 |

> 용어: **단일 상속 제약** — 자바 클래스는 부모를 **하나만** 상속할 수 있다(`extends`는 한 번만). Thread를
> 상속해 버리면 그 '한 번'을 Thread에 써버려, 정작 필요한 다른 부모(예: 도메인 클래스)를 못 물려받는다.
> Runnable은 **인터페이스**라 상속과 무관하게 여러 개 구현할 수 있어 이 제약을 피한다.

비유: Runnable 주입 = **빈 일꾼(Thread)에게 업무 지시서(Runnable)를 건넴**(같은 지시서를 여러 일꾼에게
줄 수도 있음). Thread 상속 = **일까지 직접 하는 만능 직원**(업무를 바꾸려면 직원을 새로 만들어야 함).
그래서 유연한 Runnable(+람다 + 7.8의 ExecutorService)을 표준으로 쓴다.

### 1-3. ★ start() vs run() — 초보자가 가장 많이 하는 실수
"새 스레드에서 실행하라"는 명령은 **`start()`** 다. `run()`을 직접 부르면 새 스레드가 안 생긴다.

| 호출 | 무슨 일이 일어나나 | 실행하는 스레드 |
|---|---|---|
| `t.start()` | **새 스레드를 생성**하고, 그 새 스레드가 `run()`을 실행 | 새 스레드 |
| `t.run()` (직접 호출) | 새 스레드 안 생김. 그냥 **평범한 메서드 호출** | 현재 스레드(예: main) |

왜 그런가: `run()`은 그냥 "할 일이 담긴 보통 메서드"다. 직접 부르면 다른 메서드처럼 **지금 스레드가**
그 자리에서 실행할 뿐이다. 반면 `start()`는 JVM에 "OS 수준의 새 실행 흐름을 만들어 거기서 run()을
돌려라"라고 요청하는 특별한 메서드다.
- 그래서 멀티스레드로 동시에 돌리려면 **반드시 `start()`**. `run()` 직접 호출은 동시 실행이 아니다(그냥 순차 호출).
- 확인법: `run()` 안에서 `Thread.currentThread().getName()`을 찍어 "main"이 나오면 새 스레드가 아니었다는 뜻.

### 1-4. 다 만든 스레드 다루기 — join()과 데몬 스레드
**join() — "끝날 때까지 기다리기".**
`worker.join()`을 부르면, 그 줄을 실행하는 스레드(예: main)는 **worker가 완전히 끝날(TERMINATED) 때까지
멈춰 기다렸다가** 그 다음 줄로 간다. 새 스레드의 결과를 받아 이어서 처리해야 할 때 쓴다.
- 비유: 심부름 보낸 사람(worker)이 돌아올 때까지 문 앞에서 기다리는 것.
- (이때 기다리는 main은 WAITING/TIMED_WAITING 상태가 된다 — 1-1의 대기 상태.)

**데몬 스레드(daemon thread) — "주인 따라 죽는 보조 스레드".**
스레드에는 두 종류가 있다.
- **일반(user) 스레드**: 기본값. JVM은 일반 스레드가 **하나라도 살아 있으면** 프로그램을 안 끝낸다.
- **데몬 스레드**: '보조' 스레드. JVM은 **일반 스레드가 모두 끝나면**, 데몬이 아직 일하고 있어도
  **기다리지 않고 곧장 종료**한다(데몬도 그 순간 강제로 죽는다).

그래서 데몬 스레드는 **작업을 끝까지 마친다는 보장이 없다.** 끝까지 반드시 완료돼야 하는 일(파일 저장,
로그 기록 등)은 데몬으로 만들면 안 된다. 반대로 백그라운드 모니터링·캐시 청소처럼 "프로그램 살아 있는
동안만 거들면 되는" 일에 적합하다.
- 설정: `thread.setDaemon(true)` — 단, **반드시 `start()` 호출 전에** 해야 한다(시작 후엔 못 바꿈).
- 비유: 일반 스레드 = 정직원(다 끝나야 회사 문 닫음), 데몬 = 아르바이트(정직원 다 가면 일하던 중에도 같이 퇴근).

### 실행 결과 — 가설과 실제 비교

**예시 1 (상태)** — 가설 1.
- start() 전: NEW / 실행 직후: RUNNABLE / sleep 중: TIMED_WAITING / join 후: TERMINATED. ✅
- (상태는 타이밍에 의존하므로 샘플 시점에 따라 조금 다를 수 있다.)

**예시 2 (start vs run)** — 가설 2.

| 호출 | 실행 스레드 |
|---|---|

| `MyThread().start()` | subclass-thread (새 스레드) |

→ start()만 새 스레드. run() 직접 호출은 그냥 메서드 호출. ✅

**예시 3 (join/daemon)** — 가설 3.
- join: worker가 1,2,3을 다 찍을 때까지 main 대기 후 "완료 확인". ✅

### 세 예시를 관통하는 결론
스레드는 NEW에서 시작해 RUNNABLE·대기 상태를 거쳐 TERMINATED로 끝난다(예시1). 만들 땐 Runnable(인터페이스/
람다)이 권장되고, **반드시 start()로 새 스레드를 띄워야** 한다(run() 직접 호출은 동시 실행이 아님 — 예시2).
join은 완료를 기다리는 도구이고, 데몬은 일반 스레드가 끝나면 같이 죽는 보조 스레드라 완료 보장이 없다(예시3).
이 기본기 위에서 7.4(동시성 문제)와 7.8(Executor)로 나아간다.

## 3. 자기 점검

- **Q. start()와 run()을 직접 호출하는 것의 차이는?**
  - 내 답: start()는 새 스레드를 만들어 run()을 실행하고, run() 직접 호출은 새 스레드 없이 현재
    스레드에서 메서드처럼 실행된다. 동시 실행하려면 start(). (Example2의 실행 스레드 이름)

- **Q. Runnable 구현이 Thread 상속보다 권장되는 이유 3가지는?**
  - 내 답: ①단일 상속 제약 회피 ②작업(Runnable)과 실행 수단(Thread) 분리 ③Runnable 공유로 메모리 효율.

- **Q. 데몬 스레드를 쓰면 안 되는 경우는?**
  - 내 답: 끝까지 반드시 마쳐야 하는 작업(예: 파일 저장, 로그 flush). 일반 스레드가 끝나면 JVM이
    데몬을 작업 도중 죽여버리기 때문. (Example3의 daemon이 10을 못 채우고 종료)

- **Q. `main()`도 스레드가 실행하나?**
  - 내 답: 그렇다. JVM이 'main'이라는 이름의 스레드를 자동으로 만들어 main()을 실행한다. 평소의 평범한
    코드도 사실 main 스레드 하나가 실행하는 것이고, 멀티스레드는 거기에 스레드를 더 만드는 것이다.

- **Q. main에서 `Thread.sleep(100)`을 부르면 내가 만든 worker 스레드도 같이 멈추나?**
  - 내 답: 아니다. `Thread.sleep`은 static이라 '그 줄을 실행하는 현재 스레드'만 재운다. main에서 부르면
    main만 자고 worker는 자기 일정대로 계속 흐른다. (그래서 Example1에서 main이 자는 동안 worker의 상태가 바뀐다.)

- **Q. 스레드 상태 6가지를 한 줄로?**
  - 내 답: NEW(start 전) / RUNNABLE(실행·대기) / BLOCKED(락 대기) / WAITING(무기한 대기) /
    TIMED_WAITING(시간 제한 대기) / TERMINATED(종료). (Example1)

- **Q. join()은 무엇을 누가 기다리는 것인가?**
  - 내 답: `worker.join()`을 부른 '현재 스레드'가 'worker가 끝날 때까지' 멈춰 기다린다. 새 스레드의
    작업 완료 후 이어서 처리할 때 쓴다. (Example3)
