## 1. 학습 내용 — 메서드에 인자를 넘길 때 무엇이 전달되는가

### 로우레벨부터 — C의 포인터
C에서는 `*`로 메모리 주소를 직접 다룬다. `int *p = &a;`에서 `&a`는 변수 a의 **주소**,
`*p`는 그 주소를 **역참조**한 실제 값이다. 포인터는 강력하지만 잘못 다루면 메모리 사고가 난다.
이 "주소를 다룬다"는 개념이 pass by value/reference 이해의 출발점이다.

### Pass by value vs Pass by reference
- **Pass by value (값에 의한 전달)**: 인자의 **값을 복사**해서 넘긴다. 호출된 메서드 안에서
  매개변수를 바꿔도 **원본에 영향이 없다**(복사본을 바꾼 것이므로).
- **Pass by reference (참조에 의한 전달, C++의 `&`)**: 변수 **자체의 별칭**을 넘긴다.
  메서드 안의 변경이 **원본에 그대로 반영**된다(같은 변수이므로).

### 자바는 Pass by value만 존재한다
자바에는 pass by reference가 **없다.** 항상 pass by value다. 다만 타입에 따라 "복사되는 값"이 다르다.

- **원시 타입(int 등)**: **값 자체**가 복사된다. 메서드 안 변경은 원본과 완전히 무관.
- **객체 타입**: **Heap 주소값(참조)** 이 복사된다. C 포인터와 비슷하지만 **직접 조작은 불가**하다.
  여기서 헷갈림이 생긴다 — 주소값이 복사되므로 두 갈래로 갈린다:

```java
static void modify(MyObject obj) {
    obj.value = 20;          // (a) 복사된 주소가 가리키는 '같은 객체'의 필드 변경 -> 호출자에 반영됨
    obj = new MyObject(30);  // (b) 매개변수 obj(지역 변수)에 새 객체 대입 -> 호출자와 무관
}
```

(a)는 "같은 객체를 가리키는 두 주소"라서 한쪽으로 필드를 바꾸면 둘 다 보인다.
(b)는 매개변수 obj가 메서드 스택 프레임의 **지역 변수(복사된 주소)** 일 뿐이라, 거기에 새 주소를
넣어도 호출자의 변수와는 끊긴다.

### "주소값 복사"와 "pass by reference"의 차이 (핵심)
둘 다 객체를 공유하는 것처럼 보이지만 결정적으로 다르다.
- **주소값 복사(자바)**: 같은 객체의 **필드 변경은 공유**하지만, **변수 자체의 재지정(=, swap)은
  공유하지 않는다**. 매개변수는 어디까지나 호출 메서드 프레임의 별도 지역 변수다.
- **pass by reference(C++)**: 변수 **자체가 공유**되어 재지정(swap)까지 호출자에 반영된다.

그래서 자바에서는 메서드로 두 참조를 swap해도 호출자에서는 교환되지 않는다(아래 예시 3).

### 가장 헷갈리는 지점 — "내부 수정은 밖에서도 보이는데 왜 pass by value인가?"
이게 핵심 함정이다. `changeName(member)` 안에서 `member.setName("영희")`를 하면 **메서드 밖에서도**
원본의 이름이 바뀌어 있다. "원본이 바뀌네? 그럼 reference 아냐?"라는 착각이 여기서 생긴다.

답은 **"객체를 넘긴 게 아니라, 객체로 가는 화살표(주소)를 복사해서 넘긴 것"** 이다.
- 화살표(참조)는 **2개** → 변수는 따로따로(스택에 별도 칸).
- 객체는 **1개** → 화살표 끝의 내용물은 공유.

비유하자면 **종이에 적힌 집 주소를 복사해 친구에게 준 것**이다. 친구가 그 집에 가서 가구를
바꾸면(`setName`) 내가 가도 바뀌어 있다. 하지만 친구가 자기 종이에 다른 주소를 적으면(재할당)
내 종이의 주소는 그대로다.

### 스택/힙 메모리로 시각화
스택에는 **변수(참조값=주소)**, 힙에는 **실제 객체**가 산다. 변수 → 객체로 화살표가 이어진다.

**기본 상태** `Member m = new Member("철수");`
```
        STACK                            HEAP
  ┌─────────────────────┐         ┌──────────────────────┐
  │  m  │ 0x100  ────────┼────────▶│ 0x100                │
  │     │(참조값=주소)    │         │  Member { 철수 }      │
  └─────────────────────┘         └──────────────────────┘
```

**시나리오 ① 내부 수정** `changeName(m)` → `member.setName("영희")`
호출 순간 `m`의 값(0x100)이 **복사**되어 `member`에 들어간다. 두 변수가 같은 객체를 가리킨다.
화살표를 따라가 객체 내용물을 바꾸므로 밖에서도 보인다.
```
        STACK                            HEAP
  ┌─────────────────────┐         ┌──────────────────────┐
  │ member │ 0x100  ─────┼───┐     │ 0x100                │
  │        │             │   ├────▶│  Member { 영희 } ★바뀜 │
  │  m     │ 0x100  ─────┼───┘     │                      │
  └─────────────────────┘         └──────────────────────┘
       두 변수 다 0x100 → 같은 객체 → 내용 변경은 공유됨 ✅
       호출 후 m.getName() == "영희"
```

**시나리오 ② 변수 재할당** `reassign(m)` → `member = new Member("영희")`
힙에 새 객체(0x200)가 생기고 `member` 화살표만 그쪽으로 옮겨간다. `m`은 여전히 0x100.
```
        STACK                            HEAP
  ┌─────────────────────┐         ┌──────────────────────┐
  │ member │ 0x200  ─────┼────┐    │ 0x100                │
  │ (주소 바뀜)          │    │    │  Member { 철수 }  ◀───┼─ m 은 그대로
  │  m     │ 0x100  ─────┼──┐ │    ├──────────────────────┤
  └─────────────────────┘  │ └───▶│ 0x200                │
                           └─────▶│  Member { 영희 } ★새객체│
                                  └──────────────────────┘
       member 만 0x200 으로 이동, m 은 0x100 유지 → 원본 무관 ❌
       호출 후 m.getName() == "철수"
```

**결정적 관찰**: 스택의 변수 `m`이 가진 값(`0x100`)은 **어느 경우에도 절대 바뀌지 않는다.**
메서드는 그 값의 *복사본*만 받기 때문이다 → 이것이 pass by value의 증거. 바뀌는 건 오직 "힙에 있는
객체의 내용물"뿐이고, 그건 두 화살표가 같은 곳을 가리켜 생기는 **공유 효과**일 뿐 reference가 아니다.

### pass by value vs (가정) pass by reference — 무엇을 넘기느냐
정의를 가르는 기준은 **"넘기는 게 값이냐, 변수 칸 자체냐"** 이다. `setName`(내부 수정)만 보면
두 방식의 결과가 **같아서** 구분이 안 된다. 진짜 차이는 **재할당**에서 드러난다.

| | pass by value (자바) | pass by reference (가정) |
|---|---|---|
| 넘기는 것 | 변수 **안의 값**(참조값)을 복사 | 변수 **칸 자체**(별명/alias) |
| 두 변수 관계 | 독립된 변수, 같은 객체를 가리킴 | 같은 변수 |
| 객체 내부 수정(`setName`) | 원본 반영 ✅ | 원본 반영 ✅ |
| **변수 재할당(`= new`)** | 원본 **무관** ❌ | 원본 **바뀜** ✅ |

### 실행 결과 — 가설과 실제 비교

**예시 1 (원시 타입)** — 가설 1.

| 시점 | a 값 |
|---|---|
| 호출 전 | 10 |
| `modify` 안에서 `x = 999` | (복사본만 999) |
| 호출 후 | 10 (그대로) |

→ 값 자체가 복사되므로 메서드 안 변경이 원본과 무관. ✅

**예시 2 (객체 타입)** — 가설 2.

| 동작 | 호출 후 `box.value` | 의미 |
|---|---|---|
| `mutateField(box)` (`b.value=20`) | **20** | 같은 객체의 필드 변경 → 반영 |
| `reassign(box)` (`b=new Box(30)`) | **20 (그대로)** | 매개변수 재할당 → 호출자 무관 |

→ 복사되는 건 주소값. 필드 변경은 공유되지만 재할당은 공유 안 됨 = pass by value의 증거. ✅

**예시 3 (pass by reference 아님 = 자기 점검)** — 가설 3.

| 동작 | 메서드 안 | 호출 후(호출자) |
|---|---|---|
| `swap(p, q)` | x,y 교환됨 | **p=1, q=2 (교환 안 됨!)** |
| `reassignArg(p, q)` (`a=b`) | a가 b 가리킴 | **p=1 (그대로)** |

→ 메서드 안에서는 swap이 되지만 호출자(p,q)는 그대로. pass by reference였다면 교환됐을 것이다.
이것이 "주소값 복사 ≠ pass by reference"의 결정적 증거. ✅

### 스택 프레임으로 설명 (자기 점검 답)
메서드를 호출하면 매개변수는 **그 메서드의 스택 프레임에 새로 생긴 지역 변수**다(2.3). 인자로
넘긴 값(원시 값 또는 객체 주소값)이 **그 지역 변수로 복사**되어 들어간다.

- `arg2 = arg1`이나 `swap`처럼 **매개변수 자체에 무엇을 대입**하면, 그것은 **현재 프레임의 지역
  변수**를 바꾸는 것뿐이다. 호출자의 프레임에 있는 변수는 다른 메모리라 건드릴 수 없다.
- 반면 `obj.value = 20`처럼 **복사된 주소가 가리키는 객체(Heap)의 내용**을 바꾸면, 호출자도 같은
  객체를 가리키므로 그 변경이 보인다.

즉 "프레임의 지역 변수(주소값)를 바꾸는가" vs "그 주소가 가리키는 Heap 객체를 바꾸는가"의 차이가
2.4의 모든 헷갈림을 가른다.

### 세 예시를 관통하는 결론
자바는 **언제나 pass by value**다. 원시 타입은 값을, 객체 타입은 주소값을 복사해 넘긴다.
"객체를 넘기면 다 바뀐다"는 오해는, 사실 "복사된 주소가 같은 Heap 객체를 가리켜서 필드 변경이
공유될 뿐"이라는 것이다. 매개변수 변수 자체의 재지정(swap, 재할당)은 절대 호출자에 반영되지
않는다 — 이것이 pass by reference와의 결정적 차이이며, 스택 프레임으로 깔끔히 설명된다.

## 3. 자기 점검

- **Q. "주소값이 복사된다"와 "Pass by reference"의 차이는?**
  - 내 답: 주소값 복사는 같은 객체의 필드 변경은 공유하지만 변수 자체의 재지정(swap)은 공유 안 함.
    pass by reference는 변수 자체가 공유되어 재지정까지 반영됨. (Example3의 swap 결과가 근거)

- **Q. `arg2 = arg1`(메서드 안)이 호출자에 영향 못 주는 이유를 스택 프레임으로 설명하라.**
  - 내 답: 매개변수는 호출 메서드의 스택 프레임에 복사되어 생긴 지역 변수다. `arg2 = arg1`은
    그 프레임 안의 지역 변수가 가리키는 대상을 바꿀 뿐, 호출자 프레임의 변수와는 다른 메모리라
    영향이 없다. (Example3의 reassignArg)

- **Q. (추가 실험) 배열을 메서드에 넘겨 원소를 바꾸면?**
  - 배열도 객체다. `arr[0] = 99`(내용 변경)는 반영되지만, 메서드 안에서 `arr = new int[]{...}`
    (재할당)는 호출자에 반영 안 된다. Box 대신 배열로 직접 확인하고 Example2와 같은 원리임을 정리해본다.
