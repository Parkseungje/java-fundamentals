# PART 1 — 객체지향(OOP) 기초: 1.6 추상화의 두 도구 — 추상클래스 vs 인터페이스

> 이 문서는 커리큘럼 PART 1의 소단원 중 **1.6 추상화의 두 도구**만 다룬다.
> 1.4에서 예고한 "다중 상속 금지(다이아몬드 문제)의 보완 = 인터페이스 다중 구현"이 여기서 이어진다.

## 1. 학습 내용 — 추상클래스와 인터페이스

자바에는 "공통점을 추상화해 묶는" 도구가 두 가지 있다. 둘은 비슷해 보이지만 **의도**가 다르다.

### 추상클래스 (abstract class)
- 추상 메서드(구현 없는 메서드)를 **하나라도** 가지면 클래스 자체가 `abstract`여야 한다.
- 그러면서도 **구현된 메서드도 함께 가질 수 있다** → 공통 구현을 자식에게 물려준다.
- **공통 상태(인스턴스 필드)와 생성자를 가질 수 있다.**
- **단일 상속만** 가능(클래스는 하나만 extends).
- 직접 인스턴스화 불가(`new Shape()` 컴파일 에러). 자식을 통해서만 객체가 된다.
- **의도: "is-a" 강한 관계** — Circle is a Shape. 같은 종류이면서 공통 데이터/구현을 공유하는 계층.

### 인터페이스 (interface)
- **다중 구현 가능**(`implements A, B, C`).
- 생성자가 없다(상태를 초기화할 인스턴스 필드를 가질 수 없으므로).
- 필드를 선언하면 자동으로 `public static final` **상수**가 된다.
- **의도: "can-do" 능력** — Duck can fly, Airplane can fly. 종류는 달라도 능력이 같으면 한 타입으로 묶는다.

이 다중 구현이 바로 1.4에서 예고한 **다이아몬드 문제(클래스 다중 상속 금지)의 보완책**이다.
클래스는 상태와 구현을 가지므로 둘을 동시에 상속하면 충돌하지만, 인터페이스는 (전통적으로)
구현이 없는 "약속"이라 여러 개를 합쳐도 충돌이 없다.

### Java 8 default & static 메서드 (경계가 흐려진 사건)
- **로우레벨의 불편함**: 인터페이스에 메서드를 새로 추가하면, 그 인터페이스를 구현한 **모든
  클래스가 그 메서드를 구현해야 해서 전부 깨진다.** 라이브러리 인터페이스에 메서드 하나 추가하면
  그것을 쓰던 세상의 모든 코드가 컴파일 에러가 나는 셈이다.
- **해결**: `default` 메서드 — 인터페이스가 **구현(본문)을 직접 담을 수 있게** 했다. 새 메서드를
  default로 추가하면, 기존 구현체는 그것을 오버라이드하지 않아도 default 본문을 물려받아
  **하위 호환이 유지된다.** (원하면 오버라이드도 가능)
- 결과: 인터페이스도 "구현"을 가질 수 있게 되면서 **추상클래스와의 경계가 흐려졌다.**

### 선택 가이드
- 공통 **상태 + 공통 구현**이 있는 계층 → 추상클래스
- 여러 **능력의 조합** → 인터페이스
- **둘 다 가능하면 인터페이스 우선** (다중 구현 유연성, 결합도 ↓ — Spring/JPA 모두 인터페이스 우선)

| 항목 | 추상클래스 | 인터페이스 |
|---|---|---|
| 다중 상속/구현 | ❌ (단일 상속) | ✅ |
| 구현 메서드 | ✅ | Java 8+ default만 |
| 인스턴스 필드(상태) | ✅ | ❌ (상수만 `public static final`) |
| 생성자 | ✅ | ❌ |
| 의도 | is-a (같은 종류) | can-do (능력) |

## 2. 실습으로 확인하기

> - **가설 1**: 추상클래스는 추상+구현 메서드를 함께 갖고 공통을 재사용하며, 직접 인스턴스화는 막힌다.
> - **가설 2**: 인터페이스는 다중 구현으로 능력을 조합하고, 종류가 달라도 능력이 같으면 한 타입으로 묶인다.
> - **가설 3**: default 메서드는 인터페이스에 기능을 추가해도 기존 구현체를 깨지 않는다.

### 모델 코드 (`com.study.part01_oop.s06_abstraction`)
- 추상클래스: `Shape`(추상 area + 구현 describe + 공통 name/생성자) / `Circle` / `Rectangle`
- 인터페이스: `Flyable`(default `describeFlight` 포함) / `Swimmable` / `Duck`(둘 다 구현) / `Airplane`(Flyable만)

### 예시 3개 — 각 예시가 답하는 질문

| 예시 | 답하는 질문 | 시나리오 |
|---|---|---|
| `Example1_AbstractClass` | 추상클래스가 할 수 있는/막는 것? | Circle/Rectangle을 Shape로 다루기 + `new Shape()` 불가 |
| `Example2_InterfaceMultiImpl` | 다중 구현으로 무엇을 표현? | Duck(Flyable+Swimmable) vs Airplane(Flyable) |
| `Example3_DefaultMethod` | default 메서드가 푸는 문제? | Duck(default 사용) vs Airplane(오버라이드) |

### 실행
```bash
./gradlew compileJava
java -cp build/classes/java/main com.study.part01_oop.s06_abstraction.Example1_AbstractClass
java -cp build/classes/java/main com.study.part01_oop.s06_abstraction.Example2_InterfaceMultiImpl
java -cp build/classes/java/main com.study.part01_oop.s06_abstraction.Example3_DefaultMethod
```

### 실행 결과 — 가설과 실제 비교

**예시 1 (추상클래스)** — 가설 1.

| 동작 | 결과 | 의미 |
|---|---|---|
| `circle.describe()` | 원의 넓이 = 78.53... | 물려받은 describe()가 Circle.area() 사용(동적 바인딩) |
| `rect.describe()` | 직사각형의 넓이 = 24.0 | 같은 describe()가 Rectangle.area() 사용 |
| `new Shape("도형")` | 컴파일 에러 | 추상클래스는 직접 인스턴스화 불가 |

→ 공통(name, describe)은 부모가 모아주고, 다른 부분(area)만 자식이 채운다. is-a 계층.

**예시 2 (다중 구현)** — 가설 2.

| 객체 | 가능한 능력 | 의미 |
|---|---|---|
| Duck | `fly()` + `swim()` | 두 인터페이스 구현 → 능력 조합 |
| Airplane | `fly()`만 | Swimmable 미구현 (swim() 호출은 컴파일 에러) |
| `Flyable[] {duck, airplane}` | 둘 다 `fly()` | 종류 달라도 능력 같으면 한 타입 |

→ 인터페이스는 종류(is-a)가 아니라 능력(can-do)으로 묶는다. 다중 구현이 클래스 다중 상속의 보완책.

**예시 3 (default 메서드)** — 가설 3.

| 객체 | `describeFlight()` 결과 | 의미 |
|---|---|---|
| Duck (오버라이드 안 함) | 기본 비행 설명: 무언가가 날고 있다 | Flyable의 default 본문 사용 → 안 깨짐 |
| Airplane (오버라이드) | 비행기는 제트엔진 추력으로 비행한다 | 원하면 재정의 가능 |

→ describeFlight()가 일반 추상 메서드였다면 Duck은 구현 안 해서 컴파일이 깨졌을 것이다.
default가 본문을 주므로 "선택적 구현"이 되어 하위 호환 유지. 인터페이스가 구현을 갖게 되어
추상클래스와 경계가 흐려졌다.

### 세 예시를 관통하는 결론
추상클래스(예시1)는 "같은 종류(is-a) + 공통 상태/구현"을 한 줄기 계층으로 묶고, 인터페이스(예시2)는
"종류와 무관한 능력(can-do)"을 자유롭게 조합한다. Java 8 default(예시3)는 인터페이스에 구현을
허용해 두 도구의 경계를 흐렸지만, **상태(인스턴스 필드)와 생성자는 여전히 추상클래스만 가질 수 있다**는
점이 둘을 가르는 마지막 경계다. 그래서 "공통 상태가 필요하면 추상클래스, 능력 조합이면 인터페이스,
애매하면 인터페이스 우선"이라는 가이드가 성립한다.

## 3. 자기 점검

- **Q. 추상클래스와 인터페이스, 둘 다 가능할 때 왜 인터페이스를 우선하나?**
  - 내 답: 인터페이스는 다중 구현이 되어 유연하고(한 클래스가 여러 능력 조합 가능), 구현에
    얽매이지 않아 결합도가 낮다. 단일 상속을 추상클래스에 써버리면 다른 상속 여지가 사라진다.
  - 근거: `Duck`이 Flyable+Swimmable을 동시에 가질 수 있던 것은 인터페이스라서 가능했다.

- **Q. (추가 실험) 두 인터페이스가 같은 시그니처의 default 메서드를 가지면?**
  - `Swimmable`에도 `default String describeFlight()`를 추가하고 `Duck`이 둘 다 구현하게 해본다.
    컴파일 에러(`inherits unrelated defaults`)가 나는지 확인하고, `Duck`에서 `describeFlight()`를
    직접 오버라이드해 `Flyable.super.describeFlight()`로 명시 해결하는 방법을 실험해본다.
    (인터페이스 default에서도 다이아몬드 비슷한 충돌이 생길 수 있고, 자바는 "직접 오버라이드로
    해결하라"고 강제한다 — 1.4 다이아몬드 문제와 연결)
