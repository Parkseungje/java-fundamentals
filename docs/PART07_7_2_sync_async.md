# PART 7 — 멀티스레딩과 동시성: 7.2 Sync/Async × Blocking/Non-Blocking 4분면

> 이 문서는 커리큘럼 PART 7의 소단원 중 **7.2 Sync/Async × Blocking/Non-Blocking**을 다룬다.
> (커리큘럼에서 "★ 면접 단골"로 표시된 핵심 구간) 두 축을 코드 동작으로 구분한다.

---

## 0. 들어가기 전에 — 핵심 용어
- **블로킹(blocking)**: 호출당한 함수가 일을 끝낼 때까지 제어권을 안 돌려줌 → 호출한 스레드가 멈춘다.
- **논블로킹(non-blocking)**: 일이 안 끝나도 제어권을 즉시 돌려줌 → 호출한 스레드는 계속 진행한다.
- **제어권(control)**: '다음에 실행할 코드를 정하는 권한' = 실행 흐름. 함수 호출 시 넘어가고 return 시 돌아온다.
- **동기(sync)**: 작업의 결과(완료)를 '내가 직접' 챙긴다(기다리거나 반복 확인).
- **비동기(async)**: 완료 처리를 위임하고, 끝나면 '콜백 등으로 통지'받는다(결과를 직접 안 챙김).
- **두 축은 독립**: 블로킹/논블로킹(제어권) ↔ 동기/비동기(결과 처리)는 서로 다른 질문이다. "동기=블로킹"으로 묶어 외우면 틀린다.

한 줄 그림: **블로킹/논블로킹 = '호출이 멈추나?'(제어권), 동기/비동기 = '결과를 누가 챙기나?'(완료 처리). 두 축을 곱하면 4분면이 나온다.**

---

## 1. 학습 내용 — 두 개의 독립된 축

이 둘은 자주 헷갈리지만 **서로 다른 축**이다. 같이 보면 안 되고, 따로 본 뒤 조합해야 한다.

### 축 1 — Blocking / Non-Blocking ("제어권")
**호출한 함수가 일을 끝낼 때까지 제어권을 붙잡는가**의 문제.
- **Blocking**: 일이 끝날 때까지 제어권을 안 돌려준다 → 호출자가 그 줄에서 **멈춘다**.
- **Non-Blocking**: 일이 안 끝났어도 제어권을 **즉시 돌려준다** → 호출자는 다른 일을 하거나 나중에 폴링.

### 축 2 — Sync / Async ("결과 처리")
**결과(완료)를 누가 챙기는가**의 문제.
- **Sync(동기)**: 호출한 쪽이 결과를 **직접 받아** 처리한다(반환값을 받아 다음 줄에서 처리).
- **Async(비동기)**: 호출된 쪽이 일을 끝낸 뒤 **콜백으로 알려준다** → 호출자는 결과를 직접 챙기지 않는다.

### 두 축을 합친 4분면

| | Blocking | Non-Blocking |
|---|---|---|
| **Sync** | 전통 IO (가장 단순/비효율) | NIO Polling |
| **Async** | `Future.get()` | `CompletableFuture`/콜백 (가장 효율적) |

- **Sync-Blocking**: 호출자가 멈춰 결과를 직접 받음. 가장 단순하지만 비효율(전통 IO read).
- **Sync-Non-Blocking**: 제어권은 즉시 받지만 결과는 호출자가 반복 확인(polling). (NIO non-blocking + while)
- **Async-Blocking**: 작업은 비동기로 던졌지만 결과는 `Future.get()`에서 **기다린다** → 결국 막힘.
- **Async-Non-Blocking**: 던지고 콜백만 등록 → 안 막히고 끝나면 콜백이 처리. **가장 효율적**(CompletableFuture).

핵심: **"비동기로 던졌다"고 다 좋은 게 아니다.** `Future.get()`은 Async지만 받을 때 Blocking이라 결국
호출자가 멈춘다. 콜백 기반 CompletableFuture라야 안 막히고 효율적이다.

## 1-B. 심화 정리 — 두 축의 정밀한 구분과 4분면의 흐릿한 칸들

### 두 축의 정의 (한쪽은 단단, 한쪽은 물렁)
- **블로킹/논블로킹 — 📌 정의 명확.** 논쟁의 여지가 거의 없는 단단한 축.
  - 블로킹: 호출한 함수가 일을 끝낼 때까지 제어권을 안 돌려줌 → 호출 스레드가 멈춘다.
  - 논블로킹: 일이 안 끝나도 제어권을 즉시 돌려줌 → 호출 스레드는 계속 진행한다.

  > "제어권을 안 돌려준다"의 주체는? — **호출당한 함수**다. 제어권(control) = "다음에 실행할 코드를
  > 정하는 권한", 곧 **실행 흐름** 자체다.
  > ```java
  > int n = in.read(buffer);   // 호출자(내 코드)가 read()를 호출 -> 흐름이 read() 안으로 넘어감
  > process(n);                // read()가 return해야 이 줄로 흐름이 돌아옴
  > ```
  > 함수를 호출하면 제어권이 그 함수로 넘어가고, 함수가 `return`하면 호출자에게 돌아온다.
  > - 블로킹 = 호출당한 `read()`가 일이 끝날 때까지 **return을 미뤄** 제어권을 안 돌려줌 → 호출자는
  >   `read()` 줄에 갇혀 `process`로 못 감.
  > - 논블로킹 = `read()`가 일이 안 끝나도 일단 **return(예: 0)** 해서 제어권을 즉시 돌려줌 → 호출자는 계속 진행.
  >
  > 비유: 직원(read)에게 일을 시키면 일이 끝나야 "다 했습니다"(return)하고 내가 다음 일로 간다.
  > 블로킹은 직원이 끝날 때까지 안 비켜주는 것(나는 멈춤), 논블로킹은 "아직요, 나중에 물어보세요"하고
  > 바로 비켜주는 것(나는 다른 일). (참고: read()가 데이터 도착을 기다리는 실제 일은 OS 커널에 맡기고
  > 잠든다 — 6.2/7.1. 단 '제어권 반환' 주체는 어디까지나 호출당한 함수다.)
- **동기/비동기 — ⚠️ 정의가 흐릿함.** 학계·표준에 합의된 단일 정의가 없다(완료 통지 여부 / 호출과 결과
  시점 분리 여부 / 그냥 "논블로킹"의 동의어로 쓰기도 함). 이 문서는 가장 널리 쓰이는 기준을 따른다.
  - 동기: 작업 완료를 **내가 직접** 챙긴다(기다리거나 반복 확인). 호출과 결과의 순서가 맞물린다.
  - 비동기: 완료 처리를 **위임**하고, 끝나면 콜백 등으로 **통지**받는다.

### "사분면"은 교육용 장치다
거창한 이론이 아니라, "동기=블로킹, 비동기=논블로킹"이라는 흔한 착각을 깨려고 **"두 축은 독립이다"를
보여주는 2×2 장치**다(뿌리는 유닉스 네트워크 I/O 모델 분류). 그래서 ②·③ 같은 어색한 칸의 **존재 자체가
"두 축이 따로 논다"는 증거**다 — 실무에서 깔끔히 쓰는 패턴이라기보다 반례로 끌어온 칸이라, 어색한 게 정상.

### 4분면 한눈에

| | 동기 (결과를 내가 챙김) | 비동기 (통지받음) |
|---|---|---|
| 블로킹 (호출이 멈춤) | ① `read()`, JDBC — 📌 명확 | ③ `Future.get()` — ⚠️ 흐릿 |
| 논블로킹 (즉시 리턴) | ② 논블로킹 폴링 — ⚠️ 흐릿 | ④ 콜백/이벤트 루프 — 📌 명확 |

카페 비유 (두 손잡이를 따로 돌린다):
- 손잡이 A (동기/비동기): 완성을 **내가 직접 확인**(동기) / **진동벨로 통지**받음(비동기)
- 손잡이 B (블로킹/논블로킹): 그 자리에 **묶임**(블로킹) / 자리 떠서 **다른 일**(논블로킹)

### 칸별 설명

**① 동기 + 블로킹 — 📌 명확.** 호출하면 결과가 올 때까지 스레드가 멈추고, 결과를 내가 직접 받아 다음
줄로 간다. 네 칸 중 정의 논란이 가장 적고 실무에서 가장 흔하다.
```java
int n = in.read(buffer);   // 데이터 올 때까지 이 줄에서 멈춤
process(buffer, n);
```
- `read(byte[])`: 읽은 바이트 수 반환, 데이터 없으면 올 때까지 멈춤(블로킹), EOF면 -1.
- 대표 실무: JDBC 쿼리, 전통적 Spring MVC 요청 처리.

**② 동기 + 논블로킹 — ⚠️ 관점에 따라 갈림.** 호출은 즉시 리턴(논블로킹)하지만 결과는 내가 루프로 반복
확인(동기, 폴링).
```java
ch.configureBlocking(false);
while (true) {
    int n = ch.read(buffer);        // 즉시 반환 — 데이터 없으면 0
    if (n > 0) { process(buffer); break; }
    doOtherWork();                  // 그동안 다른 일
}
```
- 흐릿한 이유: `read`만 보면 논블로킹이지만, `while`로 계속 돌리면 "결과 나올 때까지 안 놓아준다"는
  점에서 동기적으로도 보인다. 보는 각도에 따라 해석이 갈린다.

**③ 비동기 + 블로킹 — ⚠️ 관점에 따라 갈림.** 작업은 워커에 맡기지만(비동기), 결과는 `get()`으로 내가
기다려 받는다(블로킹).
```java
Future<Integer> f = ex.submit(task);  // 즉시 반환, 작업은 워커가 실행
doOtherWork();                        // get 전까진 다른 일 가능
int r = f.get();                      // 결과 아직이면 여기서 멈춤
```
- ①과의 실질 차이: 작업을 남이 하므로 여러 작업을 먼저 `submit`해두면 병렬로 돌릴 수 있다(단 submit과
  get 사이에서만 자유). 흐릿한 이유: submit 직후 바로 get하면 결과가 ①과 똑같아져, "진짜 비동기냐"는 반론이 있다.

**④ 비동기 + 논블로킹 — 📌 명확.** 작업도 맡기고 결과로 할 일까지 콜백으로 등록한 뒤 떠난다. 어디서도
멈추지 않는다. 개념적으로 가장 순수한 비동기.
```java
CompletableFuture
    .supplyAsync(() -> readFile())       // 비동기 실행, 즉시 반환
    .thenApply(data -> parse(data))      // 끝나면 변환 (콜백)
    .thenAccept(result -> log(result));  // 최종 소비 (콜백), get() 없음
doOtherWork();                           // 메인은 막힘 없이 계속
```
- 후속 콜백은 메인이 아니라 워커/이벤트 루프가 실행 → "결과 처리까지 위임" 완성형.
- 대표 실무: Node.js, Spring WebFlux/Netty, 자바 NIO2(`CompletionHandler`).

### 실무 감각
- **기본은 ①.** 직관적이고 대부분의 요청 처리에 충분. 특별한 이유 없으면 이걸 쓴다.
- **고성능·고동시성이면 ④.** 적은 스레드로 수만 동시 연결(채팅 서버, 게이트웨이, 실시간 스트리밍).
- **②·③은 특수·과도기적 패턴.** 일부러 고르기보다 라이브러리 내부나 특정 상황에서 만난다.
- ⚠️ `CompletableFuture`·`parallelStream`의 기본 공용 풀(commonPool)에 **블로킹 작업(DB·외부 API)을
  넣지 마라** — 공유 자원이라 다른 작업까지 느려진다. 전용 Executor로 분리하고, 이벤트 루프 스레드는 막지 마라.

### 정밀 요약 표

| 칸 | 호출 시 | 결과 수령 | 작업 주체 | 정의 | 대표 |
|---|---|---|---|---|---|
| ① 동기+블로킹 | 멈춤 | 내가 직접 | 내 스레드 | 📌 명확 | `read()`, JDBC |
| ② 동기+논블로킹 | 즉시 반환 | 내가 폴링 | 커널 + 내가 확인 | ⚠️ 관점차 | 논블로킹 소켓 폴링 |
| ③ 비동기+블로킹 | 즉시 반환 | 내가 `get` | 워커 스레드 | ⚠️ 관점차 | `Future.get()` |
| ④ 비동기+논블로킹 | 즉시 반환 | 콜백 통지 | 워커/루프 | 📌 명확 | `CompletableFuture`, NIO2 |

> **블로킹/논블로킹은 단단한 축**(호출이 멈추나)이고 **동기/비동기는 물렁한 축**(완료를 내가 챙기나
> 통지받나)이다. ①·④는 두 축이 같은 방향으로 정렬돼(둘 다 막거나 둘 다 안 막거나) 누가 봐도 명확하고,
> ②·③은 물렁한 축 위에 놓여 관점에 따라 해석이 갈린다.

## 2. 실습으로 확인하기

> - **가설 1**: Blocking은 호출자를 멈추고, Non-blocking은 제어권을 즉시 돌려줘 호출자가 다른 일을 한다.
> - **가설 2**: Sync는 호출자가 결과를 직접 받고, Async는 콜백이 완료 시 결과를 처리한다.
> - **가설 3**: Future.get()은 Async지만 get()에서 막히고(Async-Blocking), CompletableFuture 콜백은 안 막힌다(Async-Non-blocking).

| 예시 | 축 | 시나리오 |
|---|---|---|
| `Example1_BlockingVsNonBlocking` | 제어권 | 500ms 작업을 blocking/non-blocking 호출 |
| `Example2_SyncVsAsync` | 결과 처리 | 결과를 직접 받기 vs 콜백 |
| `Example3_FourQuadrants` | 종합 | Future.get vs CompletableFuture |


### 실행 결과 — 가설과 실제 비교

**예시 1 (제어권)** — 가설 1.
- Blocking: `blockingFetch()` 줄에서 **약 513ms 멈춤**(그동안 출력 없음) → 결과.
- Non-blocking: 호출 즉시 제어권 반환 → main이 "다른 일 하는 중..."을 5번 출력하다 완료 확인. ✅

**예시 2 (결과 처리)** — 가설 2.
- Sync: 호출자가 결과를 직접 받아 "동기 결과 처리: 데이터".
- Async: main이 "콜백 등록 후 바로 다음 일"을 먼저 찍고, 잠시 뒤 **콜백**이 결과 처리. ✅

**예시 3 (4분면)** — 가설 3.
- `Future.get()`: submit은 즉시 반환(Async)이지만 **get()에서 멈춤**(Blocking) → 결국 호출자 멈춤.
- `CompletableFuture.thenAccept()`: 호출자 안 멈추고 다음 일, 끝나면 콜백 처리(Async-Non-blocking). ✅

### 세 예시를 관통하는 결론
Blocking/Non-blocking은 "제어권을 바로 돌려주는가"(예시1), Sync/Async는 "결과를 누가 챙기는가"(예시2)로
**서로 독립된 축**이다. 둘을 합치면 4분면이 되고, 가장 효율적인 조합은 **Async-Non-blocking**(콜백 기반
CompletableFuture)이다(예시3). 흔한 오해는 "비동기면 빠르다"인데, `Future.get()`처럼 결과를 기다리면
결국 Blocking이 된다 — 진짜 비논블로킹은 콜백으로 "끝나면 알려줘"여야 한다. (Future/CompletableFuture는
7.8/7.9에서 본격적으로 다룬다.)

### ★ 예제의 `done.await()`는 데모용 장치다 (비동기의 일부가 아님)
Example2/Example3에서 콜백을 등록한 뒤 `CountDownLatch.await()`로 main을 멈추는 코드가 나온다. 이게
"비동기인데 왜 또 기다리지?" 하고 헷갈릴 수 있다.

- `await()`는 **그 자체가 블로킹**이라, 래치가 0이 될 때까지(=콜백이 `countDown()` 할 때까지) main을
  멈춘다. 그래서 그 줄 이후로 "다른 일을 못 하는" 게 아니라, **래치가 풀리면 다음 줄로 정상 진행**한다.
- **왜 넣었나?** main이 그냥 끝나면 JVM이 종료되어, 별도 스레드에서 도는 콜백이 **실행되기도 전에 죽어**
  콜백 출력을 못 본다. 그래서 "콜백이 끝날 때까지만" 붙잡아두는 **데모 전용 장치**다. 비동기 패턴 자체와는 무관하다.
- **실무에서는 이렇게 막지 않는다** — 막지 않는 것(기다리지 않는 것)이 비동기의 핵심이다.

실무에서 await 없이도 비동기가 성립하는 이유와 사례:
- **웹 서버(Spring MVC/WebFlux, Node.js)**: 서버 프로세스가 **계속 떠 있어** main이 끝날 일이 없다.
  요청을 비동기로 처리하고 콜백/`CompletableFuture`로 결과를 이어가도, 프로세스가 살아 있으니 콜백이
  반드시 실행된다. (데모처럼 await로 붙잡을 필요가 없음)
- **이벤트 루프(Netty/Node.js)**: 루프 스레드가 계속 돌며 완료 이벤트를 받아 콜백을 실행한다. 메인은
  애초에 "끝나고 기다리는" 모델이 아니라 "계속 도는" 모델이다.
- **Spring `@Async` / 메시지 큐 컨슈머**: 작업을 던지고 호출 스레드는 바로 응답을 반환한다. 결과
  처리는 풀 스레드/리스너가 맡고, 애플리케이션은 계속 살아 있어 콜백이 보장된다.
- **GUI 앱(Android/Swing)**: UI 스레드가 계속 살아 이벤트를 처리하므로, 백그라운드 작업의 콜백이
  나중에 UI 스레드에서 실행된다 — 역시 await로 막지 않는다.

> 정리: `done.await()`는 "짧게 실행되고 끝나는 main 데모"에서 콜백 출력을 보기 위한 임시 장치일 뿐이다.
> 실무 비동기 환경은 **프로세스/이벤트 루프가 계속 살아 있어** 기다릴 필요 없이 콜백이 실행된다.

## 3. 자기 점검

- **Q. Blocking/Non-blocking과 Sync/Async를 가르는 두 기준은?**
  - 내 답: Blocking/Non-blocking = "제어권을 바로 돌려주는가". Sync/Async = "결과를 호출자가 직접
    챙기는가, 콜백이 알려주는가". 둘은 독립된 축. (Example1 vs Example2)

- **Q. Future.get()은 4분면 중 어디인가? 왜?**
  - 내 답: Async-Blocking. 작업은 비동기로 던졌지만(submit) 결과는 get()에서 기다리며 호출자가 멈추기
    때문. (Example3 A)

- **Q. 가장 효율적인 조합과 그 이유는?**
  - 내 답: Async-Non-blocking(CompletableFuture 콜백). 호출자가 안 멈추고(Non-blocking) 결과는 끝났을
    때 콜백이 처리(Async)하므로 자원을 가장 잘 쓴다. (Example3 B)
