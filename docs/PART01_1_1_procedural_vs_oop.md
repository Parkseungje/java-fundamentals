## 1. 학습 내용 — 절차지향에서 객체지향으로, 왜 패러다임이 바뀌었나

### 절차지향(Procedural) 프로그래밍이란
C로 대표되는 절차지향 패러다임에서 프로그램은 **"데이터(구조체)"** 와
**"그 데이터를 처리하는 함수"** 를 별개로 다룬다. 예를 들어 자동차를 표현한다면,
`struct Car { int speed; }`라는 데이터 묶음이 있고, `accelerate(Car* car, int amount)`,
`brake(Car* car, int amount)` 같은 함수들이 그 데이터를 바깥에서 조작한다.

### 이 구조의 근본적인 불편함
이 분리 구조에서 생기는 문제는 "코드가 길다"가 아니라 **"추적성과 강제력"** 의 문제다.

1. **데이터 변경 경로를 추적하기 어렵다.**
   `Car.speed`가 바뀌는 곳이 `accelerate`인지 `brake`인지, 혹은 그 함수들을 거치지 않고
   누군가 `car->speed = 9999`로 직접 대입했는지 — 이걸 알려면 `Car` 구조체를 인자로 받는
   **모든 함수**를 찾아 다녀야 한다. 데이터와 행동이 코드베이스 전체에 흩어져 있기 때문이다.

2. **규칙(invariant)을 강제할 곳이 없다.**
   "속도는 0 미만으로 내려가지 않는다"는 규칙이 있다고 하자. 절차지향 구조에서는 이 규칙을
   `brake` 함수 안에 넣어도, 누군가 `car->speed -= 100`처럼 함수를 거치지 않고 직접 필드를
   변경하면 규칙이 무력화된다. 데이터 자체에는 "나를 이렇게 다뤄야 한다"는 정보가 없다.

3. **현실의 모델이 코드에서 흩어진다.**
   "자동차는 속도를 가지고 있고, 가속하고, 감속한다"는 하나의 개념인데,
   코드에서는 `Car`(상태)와 `accelerate`/`brake`(행동)가 서로 다른 곳에 위치한다.
   현실의 모델과 코드의 구조가 일치하지 않으니, 코드를 읽는 사람이 머릿속에서
   다시 그것들을 조립해야 한다.

### 객체지향이 제시하는 해결책
객체지향은 **"상태(필드) + 행동(메서드)을 하나의 객체 안에 묶는다"** 는 단순한 원칙으로
위 세 가지 문제를 한 번에 해결한다.

- 상태를 **private**으로 감추고(캡슐화), 그 상태를 변경하는 유일한 통로를 메서드로 제한하면
  → "데이터가 어디서 바뀌는가?"라는 질문의 답은 항상 "그 객체의 메서드 안"으로 좁혀진다.
- 규칙을 메서드 내부에 한 번만 구현해두면
  → 호출하는 모든 곳에서 규칙을 반복 검증할 필요가 없고, 빠뜨릴 수도 없다(언어가 강제).
- 상태와 행동이 한 클래스 안에 있으니
  → 현실의 "자동차"라는 개념과 코드의 `Car` 클래스가 1:1로 대응한다.

### 자주 나오는 오해 — "C 구조체로도 OOP를 흉내낼 수 있지 않나?"
가능하다. 구조체에 함수 포인터를 넣어 "객체처럼" 만들 수도 있다. 하지만 결정적인 차이는
**"그렇게 하지 않을 자유가 있는가"** 다. C에서 캡슐화·상속·다형성은 *관례*이자 *규율*이며,
컴파일러는 `car->speed = 9999`를 막을 방법이 없다. 반면 자바에서 `private` 필드는
**컴파일 타임에 언어가 강제하는 규칙**이다. "흉내 낼 수 있다"와 "언어가 보장한다"는
전혀 다른 수준의 신뢰성을 만들어낸다.

## 2. 실습으로 확인하기

위에서 정리한 이론이 맞다면, 다음과 같은 가설을 세울 수 있다:

> **가설**: 절차지향 스타일로 작성된 코드는 (1) 데이터를 우회해서 직접 변경하는 것을
> 막을 수 없고, (2) 규칙을 위반한 상태가 그대로 노출될 것이다. 반대로 객체지향 스타일은
> 이 두 가지가 코드 구조 자체에 의해 차단될 것이다.

이 가설을 실제 코드로 검증한 것이 아래 실습이다.

### 실습 코드
- `com.study.part01_oop.s01_procedural_vs_oop`
  - `CarData` + `CarProceduralOps` — **절차지향 스타일** 재현 (C 구조체 흉내: `public` 필드의
    데이터 클래스 + 그 데이터를 다루는 `static` 함수들의 분리)
  - `Car` — **객체지향 스타일** (상태를 `private`으로 감추고, 행동을 메서드로 캡슐화)
  - `ProceduralVsOopDemo` — 똑같은 "가속 50 → 브레이크 80" 시나리오를 양쪽에 적용해서
    가설이 실제로 맞는지 실행 결과로 비교

### 실행
```bash
./gradlew compileJava
java -cp build/classes/java/main com.study.part01_oop.s01_procedural_vs_oop.ProceduralVsOopDemo
```
> IntelliJ에서 ▶ 실행하면 한글 출력이 정상적으로 보인다. (Windows 콘솔 인코딩 이슈는
> `gradle.properties` / `build.gradle`의 UTF-8 강제 설정으로 처리해 두었다.)

### 실행 결과 — 가설과 실제 비교

| 검증 항목 | 가설 | 절차지향 (`CarData`/`CarProceduralOps`) | 객체지향 (`Car`) | 가설 부합 여부 |
|---|---|---|---|---|
| 규칙 위반(브레이크 과다)이 노출되는가 | 절차지향=노출, OOP=차단 | `speed = -30` 그대로 노출 | `speed = 0`으로 클램프 (`brake()` 내부 규칙) | ✅ |
| 데이터를 우회해서 직접 변경 가능한가 | 절차지향=가능, OOP=불가 | `car.speed = 9999` 컴파일·실행 모두 통과 | `car.speed = 9999` → **컴파일 에러** (`private`) | ✅ |
| 규칙이 강제되는 위치 | 절차지향=호출부마다 반복 필요, OOP=한 곳 | 없음 — `brake` 호출하는 모든 곳에서 검증 책임 | `brake()` 메서드 내부 단 한 곳 | ✅ |

→ 세 항목 모두 가설과 일치했다. 특히 "외부에서 직접 필드를 바꾸는 것"이
`CarData`에서는 **런타임에 조용히 성공**하지만 `Car`에서는 **컴파일 자체가 안 된다**는 차이가,
커리큘럼이 말하는 "흉내 낼 수는 있지만 언어가 강제하지 못한다"는 문장의 실체다.

    강제하는가"의 차이를 직접 정리해볼 것)

- **Q. (추가 실험) `CarProceduralOps.brake`에 `Math.max(0, ...)` 클램프를 추가하면
  절차지향 버전도 안전해지지 않을까?**
  - 직접 코드를 수정해 실험해보고, 그래도 남는 문제(예: 다른 함수 — `CarProceduralOps`에
    또 다른 `forceSetSpeed` 같은 함수가 추가된다면? — 가 규칙을 깜빡할 가능성)를 정리해본다.
    이것이 "캡슐화는 한 곳에 모아 강제한다"는 것의 의미다.
