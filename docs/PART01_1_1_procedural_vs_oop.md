## 0. 들어가기 전에 — 핵심 용어부터 (처음이라면 여기부터)

이 문서(그리고 PART 1 전체)에서 계속 쓰는 단어들이다. 처음이라면 먼저 익히고 내려가자.

- **패러다임(paradigm)**: 프로그램을 바라보고 짜는 '큰 사고방식'. 절차지향·객체지향이 대표적인 패러다임이다.
- **객체지향(OOP, Object-Oriented Programming)**: 프로그램을 '객체'들의 모임으로 보고, 각 객체가
  자기 상태와 행동을 책임지게 짜는 패러다임.
- **클래스(class)**: 객체를 찍어내는 '설계도/틀'. 예: `Car` 클래스 = "자동차는 이런 데이터와 기능을 가진다"는 정의.
- **객체(object) / 인스턴스(instance)**: 설계도(클래스)로 실제로 만들어낸 '실체'. `new Car()`로 만든 자동차 한 대.
  (클래스 = 붕어빵 틀, 객체 = 그 틀로 구운 붕어빵 하나.)
- **필드(field) / 상태(state)**: 객체가 가진 데이터(변수). 예: 자동차의 `speed`.
- **메서드(method) / 행동(behavior)**: 객체가 하는 일(함수). 예: 자동차의 `accelerate()`, `brake()`.
- **캡슐화(encapsulation)**: 상태(필드)를 객체 안에 감추고, 정해진 메서드로만 다루게 하는 것. 이 문서의 핵심 주제.
- **`private`**: 그 필드/메서드를 '클래스 바깥에서 직접 못 건드리게' 막는 자바 키워드. 캡슐화의 도구.
- **불변식(invariant)**: 항상 지켜져야 하는 규칙. 예: "속도는 0 미만이 될 수 없다".

한 줄 그림: **클래스(설계도)로 객체(실체)를 만들고, 객체는 필드(상태)와 메서드(행동)를 가지며,
캡슐화로 상태를 보호한다.** 이 단원은 "왜 이렇게 묶는 방식(OOP)이 그러지 않는 방식(절차지향)보다 나은가"를 다룬다.

## 1. 학습 내용 — 절차지향에서 객체지향으로, 왜 패러다임이 바뀌었나

### 절차지향(Procedural) 프로그래밍이란
C로 대표되는 절차지향 패러다임에서 프로그램은 **"데이터(구조체)"** 와
**"그 데이터를 처리하는 함수"** 를 별개로 다룬다. 예를 들어 자동차를 표현한다면,
`struct Car { int speed; }`라는 데이터 묶음이 있고, `accelerate(Car* car, int amount)`,
`brake(Car* car, int amount)` 같은 함수들이 그 데이터를 바깥에서 조작한다.

### 이 구조의 근본적인 불편함
이 분리 구조에서 생기는 문제는 "코드가 길다"가 아니라 **"추적성과 강제력"** 의 문제다.

1. **데이터 변경 경로를 추적하기 어렵다.**
   `Car.speed`가 바뀌는 곳이 `accelerate`인지 `brake`인지, 혹은 그 함수들을 거치지 않고
   누군가 `car->speed = 9999`로 직접 대입했는지 — 이걸 알려면 `Car` 구조체를 인자로 받는
   **모든 함수**를 찾아 다녀야 한다. 데이터와 행동이 코드베이스 전체에 흩어져 있기 때문이다.

2. **규칙(invariant)을 강제할 곳이 없다.**
   "속도는 0 미만으로 내려가지 않는다"는 규칙이 있다고 하자. 절차지향 구조에서는 이 규칙을
   `brake` 함수 안에 넣어도, 누군가 `car->speed -= 100`처럼 함수를 거치지 않고 직접 필드를
   변경하면 규칙이 무력화된다. 데이터 자체에는 "나를 이렇게 다뤄야 한다"는 정보가 없다.

3. **현실의 모델이 코드에서 흩어진다.**
   "자동차는 속도를 가지고 있고, 가속하고, 감속한다"는 하나의 개념인데,
   코드에서는 `Car`(상태)와 `accelerate`/`brake`(행동)가 서로 다른 곳에 위치한다.
   현실의 모델과 코드의 구조가 일치하지 않으니, 코드를 읽는 사람이 머릿속에서
   다시 그것들을 조립해야 한다.

### 객체지향이 제시하는 해결책
객체지향은 **"상태(필드) + 행동(메서드)을 하나의 객체 안에 묶는다"** 는 단순한 원칙으로
위 세 가지 문제를 한 번에 해결한다.

- 상태를 **private**으로 감추고(캡슐화), 그 상태를 변경하는 유일한 통로를 메서드로 제한하면
  → "데이터가 어디서 바뀌는가?"라는 질문의 답은 항상 "그 객체의 메서드 안"으로 좁혀진다.
- 규칙을 메서드 내부에 한 번만 구현해두면
  → 호출하는 모든 곳에서 규칙을 반복 검증할 필요가 없고, 빠뜨릴 수도 없다(언어가 강제).
- 상태와 행동이 한 클래스 안에 있으니
  → 현실의 "자동차"라는 개념과 코드의 `Car` 클래스가 1:1로 대응한다.

### 자주 나오는 오해 — "C 구조체로도 OOP를 흉내낼 수 있지 않나?"
가능하다. 구조체에 함수 포인터를 넣어 "객체처럼" 만들 수도 있다. 하지만 결정적인 차이는
**"그렇게 하지 않을 자유가 있는가"** 다. C에서 캡슐화·상속·다형성은 *관례*이자 *규율*이며,
컴파일러는 `car->speed = 9999`를 막을 방법이 없다. 반면 자바에서 `private` 필드는
**컴파일 타임에 언어가 강제하는 규칙**이다. "흉내 낼 수 있다"와 "언어가 보장한다"는
전혀 다른 수준의 신뢰성을 만들어낸다.

## 2. 실습으로 확인하기

위에서 정리한 이론이 맞다면, 다음과 같은 가설을 세울 수 있다:

> **가설**: 절차지향 스타일로 작성된 코드는 (1) 데이터를 우회해서 직접 변경하는 것을
> 막을 수 없고, (2) 규칙을 위반한 상태가 그대로 노출될 것이다. 반대로 객체지향 스타일은
> 이 두 가지가 코드 구조 자체에 의해 차단될 것이다.

이 가설을 **서로 다른 각도의 예시 3개**로 나눠 검증한다. 같은 자동차 모델을 두고,
"정상일 때 / 규칙을 어길 때 / 통로를 우회할 때"라는 세 상황을 각각 비교하면
차이가 "어디서, 왜" 생기는지가 단계적으로 드러난다.

### 공통 모델 코드 (`com.study.part01_oop.s01_procedural_vs_oop`)
- `CarData` + `CarProceduralOps` — **절차지향 스타일** 재현 (C 구조체 흉내: `public` 필드의
  데이터 클래스 + 그 데이터를 다루는 `static` 함수들의 분리). `CarProceduralOps`에는 규칙을
  지키는 `brake`와 규칙을 깜빡한 `applyPenalty`를 일부러 함께 두어 "규칙 흩어짐"을 재현한다.
- `Car` — **객체지향 스타일** (상태를 `private`으로 감추고, speed를 바꾸는 모든 메서드가
  단일 통로 `changeSpeedBy()`를 거치게 해 규칙을 한 곳에 모음).

### 예시 3개 — 각 예시가 답하는 질문

| 예시 | 답하는 질문 | 시나리오 |
|---|---|---|
| `Example1_NormalFlow` | 정상 범위에선 두 방식이 다른가? | 가속 60 → 브레이크 20 (0 밑으로 안 감) |
| `Example2_ScatteredRuleViolation` | 규칙을 어기는 입력 + 규칙이 함수마다 흩어지면? | 가속 50 → 브레이크 80 → applyPenalty 30 |
| `Example3_EncapsulationBreach` | 함수를 우회해 필드를 직접 건드릴 수 있는가? | `car.speed = 9999` 직접 대입 시도 |

### 실행
```bash
./gradlew compileJava
java -cp build/classes/java/main com.study.part01_oop.s01_procedural_vs_oop.Example1_NormalFlow
java -cp build/classes/java/main com.study.part01_oop.s01_procedural_vs_oop.Example2_ScatteredRuleViolation
java -cp build/classes/java/main com.study.part01_oop.s01_procedural_vs_oop.Example3_EncapsulationBreach
```

> `gradle.properties` / `build.gradle`의 UTF-8 강제 설정으로 처리해 두었다.)

### 실행 결과 — 가설과 실제 비교

**예시 1 (정상 흐름)** — 가설: 정상 범위에선 결과가 같다.

| | 절차지향 | 객체지향 | 가설 부합 |
|---|---|---|---|
| 최종 speed | `40` | `40` | ✅ 동일 |

→ 정상 흐름만 보면 두 방식의 차이가 드러나지 않는다. 차이는 예시 2·3의 비정상 상황에서 나온다.
이 예시는 그 "기준선" 역할을 한다.

**예시 2 (규칙 위반 + 규칙 흩어짐)** — 가설: 절차지향은 규칙을 깜빡한 함수에서 샌다.

| 단계 | 절차지향 | 객체지향 | 가설 부합 |
|---|---|---|---|
| 1단계 `brake(80)` | `0` (brake엔 클램프 있음) | `0` | ✅ |
| 2단계 `applyPenalty(30)` | **`-30`** (applyPenalty가 클램프 누락) | `0` (단일 통로가 강제) | ✅ |

→ 절차지향은 `brake`는 막았지만 `applyPenalty`에서 규칙이 새어 음수가 됐다. 규칙이 함수마다
흩어져 있기 때문이다. 객체지향은 어떤 동작을 거치든 `changeSpeedBy` 한 곳을 통과하므로 항상 0 이상.

**예시 3 (캡슐화 우회)** — 가설: 절차지향은 직접 대입을 못 막고, 객체지향은 컴파일 단계에서 막는다.

| | 절차지향 | 객체지향 | 가설 부합 |
|---|---|---|---|
| `car.speed = 9999` 직접 대입 | **런타임에 조용히 성공** → `9999` | **컴파일 에러** (`private`, 코드에 주석 처리) | ✅ |

→ "외부에서 직접 필드를 바꾸는 것"이 `CarData`에서는 런타임에 조용히 성공하지만 `Car`에서는
**컴파일 자체가 안 된다**. 이 차이가 커리큘럼이 말하는 "흉내 낼 수는 있지만 언어가 강제하지
못한다"는 문장의 실체다.

### 세 예시를 관통하는 결론
정상 흐름(예시1)에서는 차이가 없다가, 규칙을 어기는 순간(예시2) 절차지향의 "흩어진 규칙"이
새기 시작하고, 통로를 우회하는 순간(예시3) 절차지향은 아예 방어 수단이 없어진다.
객체지향은 이 모든 위협을 **상태를 감추고(private) 변경 통로를 하나로 모으는** 캡슐화로 막는다.

## 3. 자기 점검

- **Q. C 구조체로 OOP를 흉내낼 수 있지만 자바와의 결정적 차이는?**
  - 내 답: (실습 결과의 "외부에서 직접 변경" 행을 근거로, "관례에 의존하는가 vs 컴파일러가
    강제하는가"의 차이를 직접 정리해볼 것)
  - 근거 코드: `CarData.speed = 9999`(컴파일 통과) vs `Car.speed = 9999`(컴파일 에러)

- **Q. (예시 2 응용) 절차지향에서도 `applyPenalty`에 `Math.max(0, ...)`를 넣으면 되지 않나?**
  - 맞다. 하지만 그건 "함수를 하나 더 만들 때마다 개발자가 규칙을 기억해야 한다"는 뜻이다.
    `CarProceduralOps`에 speed를 깎는 4번째, 5번째 함수가 생기면 그때마다 클램프를 반복해야 하고,
    한 번이라도 빠뜨리면 예시 2의 `-30`이 재현된다. 반면 `Car`는 `changeSpeedBy` 한 곳만 지키면
    새 메서드를 아무리 추가해도 규칙이 자동으로 따라온다. → 직접 `applyPenalty`에 클램프를 넣어
    고친 뒤, "그래도 남는 구조적 위험"이 무엇인지 한 줄로 정리해본다.

- **Q. (예시 3 응용) 객체지향에서도 `getSpeed()`처럼 `setSpeed(int)`를 public으로 열면
  결국 똑같이 망가지지 않나?**
  - 그렇다 — 캡슐화는 "private 필드"만으로 완성되지 않고, **어떤 변경 통로를 외부에 열어줄지**의
    설계까지 포함한다. `setSpeed`를 무방비로 열면 예시 3의 직접 대입과 다를 바 없어진다.
    `Car`가 일부러 setter 없이 `accelerate`/`brake` 같은 "의미 있는 행동"만 노출한 이유를 정리해본다.
