# CLAUDE.md — java-fundamentals

이 프로젝트에서 작업할 때 따를 컨벤션.

## 참고 문서
전체 커리큘럼 원본은 `../master_curriculum.md` (`C:\develop\study\master_curriculum.md`)에 있다.
이 프로젝트는 그중 PART 1~7을 다룬다. PART 번호·소단원 번호·"자기 점검" 질문은
모두 이 파일을 기준으로 하며, docs/ 노트 작성 시에도 이 문서의 구조(고통→해결, 자기 점검)를 따른다.
이 파일은 두 프로젝트(java-fundamentals, spring-fundamentals)가 공유하므로 프로젝트 내부로 복사하지 않는다.

## 목적
"자바 → 스프링 통합 마스터 커리큘럼"의 PART 1~7을 코드로 직접 검증한다.
이론을 읽는 데서 그치지 않고, **터미널 도구(javap, jstack, jcmd, GC 로그 등)로 실제 동작을 눈으로 확인**하는 것이 핵심.

## 패키지 컨벤션
- `com.study.partNN_topic` — PART 번호(2자리) + 주제. 커리큘럼의 PART 번호와 정확히 일치시킨다.
- 커리큘럼이 "로우레벨의 고통 → 하이레벨 해결" 구조로 서술되는 단원은,
  가능하면 `traditional/`(고통 버전)과 `improved/`(해결 버전) 하위 패키지로 나눠 대비를 코드로 보여준다.
  예: `part08_ioc.traditional.UserDao` (Connection 직접 생성) vs `part08_ioc.improved.UserDao` (DI)

## 코드 작성 원칙
- 각 예제는 **실행 가능한 `main()` 또는 테스트**로 작성 — 눈으로 결과를 봐야 학습이 됨
- 자기 점검 질문(커리큘럼의 "자기 점검" 섹션)에 답하는 코드/실험을 우선순위로 작성

### 한 패키지(소단원)당 최소 3개 이상의 예시를 만든다
- 소단원 패키지(`s01_xxx`, `s02_xxx`...) 하나에는 **하나의 데모만 두지 말고, 서로 다른 각도에서
  같은 개념을 보여주는 예시를 최소 3개 이상** 작성한다. 같은 개념이라도 시나리오·입력 조건·
  경계 케이스를 달리하면 "왜 그런 결과가 나오는지"를 비교·대조하며 이해할 수 있다.
- 예시들은 **"비교 가능하도록"** 설계한다 — 같은 구조를 다른 조건에 적용하거나(정상 케이스 /
  경계 케이스 / 실패 케이스), 같은 문제를 다른 방식으로 푼 버전을 나란히 둔다.
  예: PART 1.1이라면 ①정상 동작 시나리오 ②규칙을 위반하는 경계 케이스
  ③외부에서 캡슐화를 우회하려는 시도, 이렇게 세 갈래로 나누면 "차이점이 어디서, 왜 발생하는지"가
  뚜렷해진다. 단순히 비슷한 예제를 양적으로 늘리는 게 아니라, **각 예시가 서로 다른 질문에
  답하도록** 설계하는 것이 핵심이다.
- 예시 파일명은 의도가 드러나게 짓는다 (예: `Example1_NormalFlow`, `Example2_BoundaryCase`,
  `Example3_EncapsulationBreach`처럼 — 단순 번호보다 "무엇을 보여주는 예시인지"가 드러나야 함).

### 코드 내부 주석은 상세하게 적는다
- 주석은 "왜 이렇게 동작하는지·왜 이렇게 설계했는지"에 집중하되, **분량을 아끼지 않는다.**
  특히 다음을 반드시 포함한다:
  - 이 코드가 커리큘럼의 어떤 개념·이론을 보여주기 위한 것인지
  - 왜 이런 입력값/시나리오를 선택했는지 (예: "80을 뺀 이유 — 가속한 50보다 크게 해서 음수
    경계를 일부러 넘기기 위함")
  - 이 코드를 실행하면 어떤 결과가 "왜" 나와야 하는지에 대한 예측
  - 비교 대상 예시와 무엇이 다른지에 대한 명시적 언급 ("Example1과 달리 여기서는...")
- 즉 주석만 읽어도 "이 코드가 무엇을 입증하려고 하는지"와 "그 근거가 무엇인지"를
  알 수 있어야 한다. 짧은 한 줄 주석("// 가속")보다, 그 결정의 배경과 기대 결과를
  설명하는 주석("// 가속 후 브레이크를 더 크게 걸어 음수 경계를 넘기는지 확인 — 절차지향
  버전은 이를 막을 수단이 없으므로 -30이 그대로 노출될 것으로 예상")을 지향한다.

## docs/ 작성 규칙
- 파일명: `PARTNN_M_M_주제.md` (소단원 단위, 예: `PART01_1_1_procedural_vs_oop.md`)
- **이 문서는 "코드 사용법 정리"가 아니라 학습 자료다.** 순서를 반드시 지킨다:
  1. **먼저 학습 내용(이론)을 적는다** — 커리큘럼의 해당 소단원이 설명하는 개념·배경·"왜"를
     충분히 풀어서 정리한다(요약 한 줄이 아니라, 그 개념을 모르는 사람도 이해할 수 있는 설명).
  2. **그다음 그 개념을 실습으로 어떻게 확인하는지** 연결한다 — "이 이론이 맞다면, 코드에서
     이런 결과가 나와야 한다"는 가설 → 실행 → 실제 결과 비교의 흐름으로 서술한다.
  3. 직접 실행해서 확인한 결과(출력, javap, GC 로그, 스레드 덤프 등)와
     자기 점검 질문에 대한 답, 예상과 다르게 동작한 부분을 기록한다.
- 즉 "이 코드는 이렇게 동작합니다"가 아니라 **"커리큘럼은 이렇게 설명한다 → 그래서 이 코드를
  실행하면 이런 결과가 나와야 하고, 실제로 이렇게 나왔다"** 순서로 쓴다. 이론 없이 실습 결과만
  나열하는 문서는 이 프로젝트의 목적(학습 자료)에 맞지 않는다.

## 자바 버전 주의
- 이 프로젝트는 **Java 21** toolchain으로 고정되어 있다 (`build.gradle`의 `JavaLanguageVersion.of(21)`).
- 커리큘럼 예제 중에는 **자바 버전에 따라 문법·동작이 달라지는 부분**이 섞여 있다
  (예: `instanceof` 패턴 매칭은 Java 16+, 가상 스레드는 Java 21+, switch 패턴 매칭 등).
- 새 예제 코드를 작성할 때는 "이 기능이 어느 버전부터 가능한지"를 확인하고,
  버전 의존적인 내용이면 코드 주석 또는 docs에 **"Java 21 기준"**임을 명시한다.
  다른 버전으로 검증해야 하는 경우 toolchain 값을 바꿔 별도로 실행하고 차이를 기록한다.
